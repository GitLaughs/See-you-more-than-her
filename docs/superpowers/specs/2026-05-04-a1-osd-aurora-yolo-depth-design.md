# A1 OSD 诊断与 Aurora YOLO/深度预览设计

## 背景
当前需要同时解决三件事：
1. 判断板端没有显示 OSD 检测框，到底是模型没有识别到目标，还是 OSD 绘制链路有误。
2. 将板端 OSD 检测框统一改为灰度绘制，避免类别颜色、LUT 或颜色映射干扰排查。
3. 复用现有 Aurora 深度图模块，把板端伪深度图改成随机深度图，并将前端深度预览改成彩色热力图；同时在 Aurora 的“A1 识别状态”模块输出 YOLO 识别结果摘要。

本次设计限制如下：
- 不重构整体协议。
- 不新增独立深度预览模块。
- “A1 识别状态”仅输出 YOLO 结果摘要，不扩成完整调试面板。
- 优先改 `ssne_ai_demo/` 和 `tools/aurora/`，不扩散到 vendor 区域。

## 目标

### 1. OSD 问题可归因
板端日志需要能明确区分以下两类情况：
- YOLO 推理结果为空，没有检测到目标。
- YOLO 已输出检测框，但 OSD 绘制或 flush 链路有问题。

### 2. OSD 框统一灰度
无论目标类别为何，板端绘制时统一使用灰度框。如果底层接口使用 RGB，则设置 `R=G=B`；如果已有单值或调色板索引接口，则选定单一灰色。

### 3. Aurora 端深度预览更直观
复用现有深度图链路，把板端伪深度图改成随机深度图，让前端预览画面持续变化；前端把深度值映射为彩色热力图，快速确认链路活跃与渲染正确。

### 4. Aurora 输出 YOLO 状态摘要
Aurora 的“A1 识别状态”模块显示当前 YOLO 摘要信息：目标数、类别名、置信度。只展示摘要，不展示完整调试细节。

## 方案

### 一、板端 OSD 诊断证据点
在最小必要位置加入 stdout 证据，不做大面积调试打印。

证据点分为三层：
1. **检测输出层**：记录当前帧检测数量；若非空，记录首个框类别、置信度、坐标。
2. **可视化输入层**：记录传给 OSD 绘制函数的框数量，确认检测结果是否成功进入绘制阶段。
3. **OSD 刷新层**：在 texture/graphic layer flush 前后记录是否执行、返回状态或关键分支是否命中。

通过这三层证据，得到以下判定：
- 检测输出为 0：优先判定为模型未识别到目标。
- 检测输出非 0，但可视化输入为 0：问题在检测到绘制之间的数据传递。
- 可视化输入非 0，但 flush 未执行或失败：问题在 OSD 绘制链路。
- flush 成功但 Aurora/板端仍看不到：再考虑镜像未更新、资源遮挡或显示层问题。

### 二、灰度框绘制策略
所有检测框采用统一灰度值绘制，不再按类别选择不同颜色。

约束：
- 不改检测类别逻辑。
- 不改框几何、坐标变换或优先级逻辑。
- 仅替换颜色来源。

实施方式：
- 如果绘制调用使用 RGB 颜色对象，则改为固定灰度，如 `(160,160,160)` 或其他对比度足够的灰色。
- 如果绘制调用走 LUT 或 palette index，则选定单一灰色对应值，并对全部类别复用同一值。

### 三、随机伪深度图策略
当前伪深度图改为随机数深度图，用于联调链路验证，不追求物理真实性。

设计要求：
- 数据范围保持在现有前端预期范围内，避免前端归一化或解码逻辑失效。
- 数据形状、尺寸、传输接口保持不变，避免协议连锁改动。
- 随机图应肉眼可见变化，避免与静态图混淆。

推荐实现：
- 在现有伪深度生成位置，按像素填充伪随机深度值。
- 如担心每帧完全随机过于闪烁，可加入简单空间结构（如行列梯度 + 随机扰动），但不作为本次必需项。

### 四、Aurora 深度预览改彩色热力图
不新增模块，直接修改现有深度预览渲染逻辑。

设计要求：
- 保持原有数据入口、刷新节奏、UI 布局不变。
- 将深度值先归一化，再映射到热力图色带。
- 颜色映射应明显区分近远变化，便于肉眼验证。

推荐渲染方式：
- 前端 canvas 或现有图像渲染路径中，将单通道深度值映射为热力图颜色。
- 可采用简单多段渐变，例如：蓝 → 青 → 绿 → 黄 → 红。
- 不必引入额外前端库。

### 五、Aurora “A1 识别状态”摘要
在现有“A1 识别状态”模块内展示 YOLO 结果摘要。

显示字段：
- 当前目标数
- 每个目标的类别名
- 每个目标的置信度

约束：
- 不增加完整调试日志区。
- 不展示底盘决策、推理耗时、时间戳等扩展字段。
- 若无检测结果，明确显示“0 个目标”或等价空状态。

数据接入策略：
- 优先复用现有 Aurora 与板端之间已有状态回传或文本解析通道。
- 仅在必要处补充 YOLO 摘要字段，不另起全新协议层。

## 数据流
1. 板端 YOLO 完成推理，得到 boxes。
2. 板端输出最小诊断日志，记录检测数量与关键框信息。
3. 板端将 boxes 传入 OSD 绘制路径，使用固定灰度值绘制。
4. 板端在深度图输出路径生成随机伪深度数据，沿现有链路发送。
5. Aurora 接收深度数据后，用现有深度图模块做彩色热力图渲染。
6. Aurora 接收或解析 YOLO 摘要后，更新“A1 识别状态”模块。

## 错误处理
- 若板端检测结果为空，前端状态模块显示空结果，不伪造历史结果。
- 若 Aurora 收到深度数据但格式异常，保留现有错误提示或空白回退，不新加复杂 fallback。
- 若 YOLO 摘要字段暂时缺失，状态模块应显示无数据，而不是阻断页面其他功能。

## 验证方案

### 板端验证
1. 编译通过。
2. 运行后日志能看到：检测数量、绘制输入数量、flush 执行证据。
3. 检测到目标时，OSD 框以灰度显示。
4. 无目标时，日志明确显示 0 boxes。

### Aurora 验证
1. Companion 页面正常打开。
2. 现有深度图模块仍能收到数据。
3. 深度图预览显示为彩色热力图，而非单色灰度图。
4. “A1 识别状态”显示 YOLO 摘要；无目标时显示空状态。

## 不做内容
- 不重构 Aurora 整体页面结构。
- 不统一检测框、深度图、调试元数据为新协议。
- 不把随机深度图做成真实深度估计。
- 不扩展“A1 识别状态”为全量调试面板。

## 实施顺序建议
1. 先定位板端检测输出与 OSD 绘制证据点。
2. 再把 OSD 颜色改为灰度。
3. 再修改随机深度图输出。
4. 然后修改 Aurora 深度图热力图渲染。
5. 最后补“A1 识别状态”YOLO 摘要展示。

## 成功标准
- 能明确回答“没框是没识别到，还是画框代码有误”。
- OSD 检测框已统一为灰度。
- Aurora 深度图预览已改为彩色热力图。
- Aurora “A1 识别状态”能展示 YOLO 结果摘要。
