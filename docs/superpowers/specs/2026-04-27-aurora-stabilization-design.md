# Aurora 启动稳定性、COM13 终端修复与商用品质前端重绘设计

- 日期：2026-04-27
- 范围：`tools/aurora/`
- 目标页面：`tools/aurora/templates/companion_ui.html`
- 目标启动入口：`tools/aurora/launch.ps1`

## 1. 目标

本次工作要同时解决三个问题：

1. 查清 `tools/aurora/launch.ps1` 启动后，A1 摄像头在 Qt bridge 连接数秒后仍无输入的根因。
2. 修复 COM13 终端显示中的错误换行和乱码问题。
3. 将 Aurora Companion 前端重绘为偏商用成品的调试前端，在不破坏现有能力的前提下提升信息层级、可读性和稳定感。

成功标准：

- 优先实现“仅靠 `launch.ps1` 就能稳定出画面”。
- 如果受厂商驱动或设备初始化机制限制，允许降级为“首次需要外部 Aurora.exe 初始化，之后 Companion 自动接管”，但必须有明确检测、提示和可重复流程。
- COM13 终端对中文、JSON、命令回显、分包日志的显示稳定，不再出现系统性错行和显著乱码。
- 前端页面采用视频优先的商用调试台布局：预览成为主工作面，控制区与终端区层级清晰，并完成浏览器实测。

## 2. 现状观察

### 2.1 启动与视频链路

当前 `launch.ps1` 仅完成 UTF-8 控制台初始化、端口释放、启动浏览器和运行 `aurora_companion.py`。A1 视频能否正常工作取决于 `aurora_companion.py` 与 `qt_camera_bridge.py` 的协作。

从代码上看：

- `aurora_companion.py` 维护 `camera_source_global`、`QT_BRIDGE_URL`、`QT_BRIDGE_PROTOCOL_VERSION` 等状态，并负责 Flask 页面的视频源暴露与状态管理。
- `qt_camera_bridge.py` 使用 `PySide6.QtMultimedia` 枚举和打开摄像头，通过本地 HTTP 暴露状态和 JPEG 帧。
- 设备识别明显依赖启发式：设备名包含 `Smartsens` / `FlyingChip` / `SC132`，或像素格式命中灰度/RAW/YUV 特征。
- A1 设备和普通 Windows 摄像头的选择、像素格式选择、分辨率/FPS 决策都带有一定推断逻辑，因此“桥接已连接但无帧”极有可能发生在设备选型、像素格式、Qt 相机状态或帧消费边界。

### 2.2 COM13 终端链路

当前 `serial_terminal.py` 已经做了若干抗分包处理：

- `_normalize_text()` 先尝试 `utf-8`，再尝试 `gb18030`。
- `_split_decodable_prefix()` 尝试找出可安全解码前缀，避免半个 UTF-8 字符立即解码。
- `_pop_complete_lines()` 同时识别 `\n`、`\r\n` 和裸 `\r`。
- `_rx_worker()` 会读串口数据，优先输出完整行，长时间空闲后才强制冲刷部分缓冲。

这说明问题未必只在后端字节解码；也可能发生在：

- A1 返回流里存在控制字符或 ANSI 片段；
- JSON、日志和回显混流导致前端按“消息”而非“真实行”渲染；
- 前端终端组件对 `white-space`、换行拼接、HTML 转义或增量刷新的处理有误。

### 2.3 前端结构与视觉现状

当前 `companion_ui.html` 已有较多功能，但设计方向仍偏“开发中控制面板”：

- 使用 Google Fonts 中的 `Space Grotesk`，视觉风格偏互联网 dashboard，不够像稳定的设备调试前端。
- 页面结构较散，信息区、控制区、视频区、终端区之间的主次关系不够强。
- 背景、边框、阴影、卡片节奏比较轻，缺少“仪器面板式”的一致性。

## 3. 调试与改造原则

1. **先找根因，再改实现。** 不先猜结论，不用“堆补丁”方式碰运气。
2. **每条链路先建立观测点。** 先证明断在何处，再做最小修复。
3. **优先保持现有能力和接口。** 不重写通信架构，不无关重构。
4. **前端改版只调整结构、视觉和交互细节，不改变主要操作路径。**
5. **最终结论必须来自本地实际运行与浏览器查看，而不是仅代码静态判断。**

## 4. 方案设计

### 4.1 A1 摄像头稳定化方案

把问题拆成五个断点逐层验证：

1. `launch.ps1` 是否确实以期望 source 启动 Companion。
2. `aurora_companion.py` 是否真的请求了 Qt bridge，并拿到了 bridge 状态。
3. `qt_camera_bridge.py` 是否成功枚举到目标设备、选择到合理格式、进入产帧状态。
4. Companion 是否定期收到 bridge 输出的帧与状态。
5. 前端 `<img>` / MJPEG 展示链路是否只是显示失败，而不是后端完全没帧。

为此将加入面向调试的最小化观测能力：

- 在 `aurora_companion.py` 内补充更明确的 bridge 启动、请求、超时、状态转换日志。
- 在 `qt_camera_bridge.py` 内暴露更完整的当前相机状态：设备名、像素格式、最近一帧时间戳、帧计数、错误状态、上次状态消息。
- 在前端增加更清楚的相机诊断区，能直接看到“已连接但无帧”“设备未枚举”“桥接已启动但未出流”等状态。

#### 根因候选

优先验证以下几类根因：

1. **Qt bridge 选错设备或选错像素格式**：启发式识别命中错误设备，或 A1 在当前驱动下需要不同格式组合。
2. **设备被独占或初始化顺序异常**：Aurora.exe 曾经先完成唤醒/初始化，而 Companion 直接打开时失败。
3. **Qt 已产帧但 Companion 取流失败**：桥接进程正常，但本地 HTTP 拉取逻辑、超时或协议版本不匹配。
4. **后端已有帧但前端未显示**：MJPEG 刷新、缓存、页面状态同步存在问题。

#### 目标行为

- 首选：`launch.ps1` 直接完成桥接拉起、设备识别和前端出图。
- 次选：若必须先经 Aurora.exe 初始化，则在 Companion 中检测该条件并给出明确引导，且初始化后可自动恢复接管，不再停留在“等待 5 秒无画面”的灰色状态。

### 4.2 COM13 终端修复方案

COM13 问题按“字节层 → 行层 → 消息层 → 视图层”四层排查。

#### 后端排查重点

- 检查 `_pop_complete_lines()` 对 `\r`、`\n`、`\r\n` 的切分是否和设备真实输出一致。
- 检查 `_normalize_text()` 是否因为 utf-8 / gb18030 回退导致同一缓冲区在不同批次被不一致解释。
- 检查 `_flush_partial_buffer()` 是否在空闲冲刷时把半行误判为完整文本，造成视觉断裂。
- 检查 `relay_comm.py` 发命令与终端回显混在同一时间窗口时，前端是否误把多条消息拼成一条或拆成多条。

#### 前端排查重点

- 终端区域是否把 `textContent` / `innerHTML` 混用，导致转义与换行语义不一致。
- CSS 的 `white-space`、`word-break`、滚动容器策略是否把单条日志折行成“错行”。
- 是否存在把 `partial=true` 的数据当完整消息渲染的逻辑。

#### 目标行为

- 一条串口逻辑消息只显示一次。
- 行尾处理一致，避免空行倍增和中间断行。
- 字节无法完全确定编码时，优先不损坏结构，再尽量保全文本内容。
- 前端使用等宽终端视觉与稳定换行策略，保证日志、JSON、十六进制显示都可读。

### 4.3 商用品质调试前端重绘方案

前端方向定为：**深色、克制、偏工业设备控制台的商用调试前端**。

不做炫技展板，而是强调“状态可信、操作稳定、信息层次清楚”。

本轮采用 **视频优先** 方案：把预览区做成主工作面，控制区收敛到右侧纵向工作栏，终端区下沉为独立控制台区域。

#### 新的信息架构

页面重组为三层：

1. **顶部全局状态栏**
   - 当前视频源
   - Qt bridge / 相机状态
   - COM13 连接状态
   - Relay 状态
   - ROS 状态
   - 当前端口与可用的小型实时指标（如 FPS / latency / frame age，能拿到才显示）
   - 关键动作入口

2. **主工作区**
   - 左侧主预览区（约 65–70%）：视频预览、检测预览、截图、源切换、空态/加载态/错误态
   - 右侧控制工作栏（约 30–35%）：连接与来源、链路检查 / relay 状态、运动控制、遥测摘要、次级设置

3. **底部终端区**
   - A1 调试终端
   - TX / RX 分视图或合并视图切换
   - 最近关键返回内容
   - 固定操作行：连接、自动连接、清空、发送

#### 终端区重点要求

- 采用深色高对比等宽控制台视觉
- 强化 TX / RX 区分与 partial 标记可见性
- 固定高度与稳定滚动策略，避免整块区域频繁闪烁
- 优先保留现有 backend 文本语义，不改协议，不重定义消息结构
- 前端渲染策略以“稳定刷新观感”为目标，避免每次轮询都造成大面积重排

#### 视觉系统

- 主体采用深色背景与较低反光卡片，不再使用当前偏浅色玻璃态方案。
- 字体以稳重的中文无衬线和等宽字体组合为主，弱化展示型字体的存在感。
- 颜色语义收敛：蓝色表示系统主状态，绿色表示就绪/正常，橙色表示等待/降级，红色表示故障。
- 状态标签、设备信息条、终端、控制按钮风格统一，形成完整调试台语言。
- 减少零散装饰色、混杂边框和不一致卡片高度，建立更强的视觉节奏。

#### 功能保持不变的约束

- 保留现有 Flask 路由和 JS 功能入口，避免大规模后端联动变更。
- 先在现有 HTML 模板上重组结构与样式，必要时整理前端脚本，但不引入新前端框架。
- 优先保留现有 DOM id 和主交互入口，避免因重排破坏事件绑定。

## 5. 测试与验证

### 5.1 A1 摄像头

- 使用 `tools/aurora/launch.ps1` 启动。
- 观察 Companion 日志、Qt bridge 状态、页面相机状态区和浏览器视频显示。
- 分别验证 `--source auto`、`--source a1`、`--source windows`。
- 如果需要 Aurora.exe 初始化，验证是否能稳定复现，以及初始化后是否能自动接管。

### 5.2 COM13 终端

- 连接 COM13，发送常见 A1_TEST 命令，如 `help`、`status`、`A1_TEST debug_status`、`A1_TEST test_echo`。
- 检查中文、JSON、回显、空闲后续包、长行文本、混合日志的显示。
- 同时比对后端日志对象与前端界面内容，确认不是单纯渲染问题。

### 5.3 前端页面

- 本地打开 Aurora 页面，检查首屏布局、滚动行为、终端区、视频区、按钮状态、响应式压缩表现。
- 验证页面在功能不退化的前提下更清晰、更像成品工具而非临时页面。

### 5.4 低成本代码校验

至少执行：

```bash
python -m py_compile tools/aurora/aurora_companion.py tools/aurora/serial_terminal.py tools/aurora/relay_comm.py tools/aurora/qt_camera_bridge.py
```

如果前端脚本被明显调整，还要通过实际浏览器加载确认无运行时错误。

## 6. 实施顺序

1. 为 A1 摄像头链路补充观测点并复现问题。
2. 确认根因后做最小修复，优先争取 `launch.ps1` 直接可用；若受初始化约束，则固化已接受的 Aurora.exe 预初始化降级流程。
3. 为 COM13 终端链路补充观测点，复现并修复错行/乱码。
4. 在功能稳定基础上，按视频优先方案重绘 `companion_ui.html`，同步调整必要的前端脚本与样式。
5. 实际启动页面、串口和视频链路做联调验证。

## 7. 非目标

本次不做以下事项：

- 不重写 Aurora 为新的前后端架构。
- 不改动 ROS2、板端 SDK 或 STM32 固件逻辑。
- 不引入新的桌面端 UI 框架。
- 不在未证明必要的情况下修改 COM 协议语义。

## 8. 风险与应对

### 风险 1：A1 相机问题源于厂商驱动或私有初始化流程

应对：保留证据链，给出自动检测和降级流程，而不是继续猜测式修补。

### 风险 2：COM13 显示问题是前后端共同造成

应对：同时记录原始字节、后端规范化文本和前端最终渲染结果，明确责任边界。

### 风险 3：前端重绘影响原有脚本事件绑定

应对：优先保留 DOM id 和主交互入口，只重组结构；变更时同步浏览器回归验证，并重点检查 `stopRosMotion`、`loadRosStatus` 等现有入口在新布局中的引用完整性。
