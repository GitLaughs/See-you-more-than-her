# Aurora / PC / A1 工具拆分设计

- 日期：2026-04-30
- 范围：`tools/aurora/`、`tools/PC/`、`tools/A1/`
- 目标：把当前 Aurora 中混合的视频、COM13、直连 STM32、A1 中继控制、ROS 调试拆成三套独立工具，避免端口、状态、日志、运行时职责串台。

## 1. 目标

本次拆分目标：

1. 保留 `tools/aurora/` 作为视频与 COM13 调试工具。
2. 新建 `tools/PC/`，承载电脑直连 STM32 与 ROS 调试能力。
3. 新建 `tools/A1/`，承载 COM13 → `A1_TEST` → STM32 的 A1 中继控制能力。
4. 三套工具运行时完全独立，各自拥有自己的 Python 入口、PowerShell 启动脚本、HTML 页面、状态文件和默认端口。
5. 拆分过程中不把当前 Aurora 的 Qt 桥连环重试、启动刷屏日志、摄像头连接不稳定问题复制到新工具中。

成功标准：

- Aurora 页面仅保留视频、拍照、COM13 终端、`A1_TEST` 手动命令和电脑-A1-STM32 联通测试。
- `tools/PC/` 页面只处理电脑直连 STM32 与 ROS 调试，不出现 COM13 终端和 A1 中继语义。
- `tools/A1/` 页面只处理 COM13 经由 A1 的中继控制，不出现直连 STM32 端口选择和 ROS 调试。
- Aurora、PC、A1 三套工具可以同时启动，默认端口固定且互不冲突。
- 三套工具不共享前端“当前目标”切换状态，不共享运行时状态文件，不互相调用对方 HTTP 接口。

## 2. 现状与问题

当前 `tools/aurora/` 同时承担了多种职责：

- Qt 摄像头桥与视频预览
- 拍照与视频流状态展示
- COM13 串口终端与 `A1_TEST` 命令入口
- 电脑直连 STM32 的底盘控制与信息读取
- COM13 经由 A1 的中继底盘控制
- ROS bridge 状态与调试入口

这种混合结构带来四类问题：

1. **职责混合**：视频调试、COM13 调试、直连底盘控制、A1 中继控制、ROS 调试堆在同一进程和同一页面。
2. **状态串台**：前端存在目标切换语义，用户容易在错误场景下对错误链路发命令。
3. **端口串台**：不同控制路径共享同一套运行时状态和页面入口，容易把 COM13 与直连 STM32 操作混用。
4. **故障扩散**：Aurora 的相机/Qt 桥启动问题会污染并拖累本该独立的底盘调试能力。

因此，本次不是在 Aurora 内继续加模式，而是把职责边界直接拆开。

## 3. 总体方案

拆成三套独立工具：

### 3.1 `tools/aurora/`

Aurora 只保留：

- 视频预览
- 拍照
- COM13 终端日志
- `A1_TEST` 手动命令输入
- 电脑-A1-STM32 联通测试

Aurora 不再保留：

- 直连 STM32 前后左右控制
- 直连 STM32 底盘信息模块
- COM13 中继底盘前后左右控制
- COM13 中继底盘信息模块
- ROS 调试区
- 前端“当前目标”切换逻辑

### 3.2 `tools/PC/`

PC 工具只保留电脑直连 STM32 与 ROS 调试：

- 直连 STM32 串口连接/断开
- 联通测试
- 前后左右控制
- 底盘信息读取
- ROS 状态与调试

PC 工具不保留：

- 摄像头与 Qt 桥
- COM13 终端
- `A1_TEST` 手动命令入口
- A1 中继控制逻辑

### 3.3 `tools/A1/`

A1 工具只保留 COM13 → `A1_TEST` → STM32 的中继能力：

- COM13 会话检查
- `A1_TEST` 命令发送
- 底盘联通测试
- 前后左右控制
- 底盘信息读取

A1 工具不保留：

- 摄像头与 Qt 桥
- 直连 STM32 端口选择
- ROS 调试

## 4. 目录与文件拆分

三套工具都必须独立拥有 Python、ps1、html。

### 4.1 Aurora 保留文件

Aurora 保留并继续精简：

- `tools/aurora/aurora_companion.py`
- `tools/aurora/qt_camera_bridge.py`
- `tools/aurora/serial_terminal.py`
- `tools/aurora/launch.ps1`
- `tools/aurora/templates/companion_ui.html`

必要时 Aurora 内可删除或停止引用底盘控制/ROS 相关模块。

### 4.2 新建 `tools/PC/`

建议结构：

- `tools/PC/pc_tool.py`：Flask 入口，注册页面与各子模块蓝图
- `tools/PC/pc_chassis.py`：直连 STM32 串口、联通测试、运动控制、底盘信息
- `tools/PC/pc_ros.py`：ROS 环境检测、状态查询、调试入口
- `tools/PC/launch.ps1`：PC 工具启动脚本
- `tools/PC/templates/index.html`：PC 工具页面

### 4.3 新建 `tools/A1/`

建议结构：

- `tools/A1/a1_tool.py`：Flask 入口，注册页面与各子模块蓝图
- `tools/A1/a1_relay.py`：COM13 → `A1_TEST` → STM32 的中继控制与联通测试
- `tools/A1/a1_status.py`：底盘信息查询与状态接口
- `tools/A1/launch.ps1`：A1 工具启动脚本
- `tools/A1/templates/index.html`：A1 工具页面

### 4.4 拆分原则

- 不是复制整份 Aurora 改目录名。
- 是按职责迁移代码。
- `tools/PC/` 与 `tools/A1/` 完全独立，不额外抽共享模块层。
- 每个工具只维护自己场景下需要的最小模块集合。

## 5. 运行时隔离设计

### 5.1 固定默认端口

三套工具默认端口固定：

- Aurora：`6201`
- PC：`6202`
- A1：`6203`

目标是允许三者同时运行，不再靠“动态抢一个空闲端口”来回避结构问题。

### 5.2 状态文件隔离

每个工具必须使用各自独立的状态文件、PID 文件、日志前缀和临时文件名。

例如：

- Aurora 使用自己现有 Qt 桥 owner state / 视频状态文件
- PC 使用独立串口连接状态文件
- A1 使用独立 COM13 / relay 状态文件

禁止三套工具共用一个“当前连接目标”或“当前控制模式”状态文件。

### 5.3 前端状态隔离

- Aurora 页面不再存在“直连 STM32 / COM13 经由 A1”目标切换按钮。
- PC 页面永远表示“电脑直连 STM32”。
- A1 页面永远表示“COM13 经由 A1”。

用户进入哪个页面，就处于哪个职责边界里，不再依赖页面内切换目标。

### 5.4 启动链路隔离

- Aurora 的 `launch.ps1` 仍负责视频相关启动。
- PC 的 `launch.ps1` 不启动 Qt 桥，不探测摄像头。
- A1 的 `launch.ps1` 不启动 Qt 桥，不探测摄像头。

PC/A1 两个新启动脚本必须保持安静、职责单一，只关心本工具自己的后端。

## 6. API 与数据流边界

### 6.1 Aurora API 边界

Aurora 后端只暴露三类路由：

1. 视频/拍照
2. COM13 终端日志
3. `A1_TEST` 手动命令与 COM13 联通测试

Aurora 不再暴露底盘方向控制、底盘信息和 ROS 调试路由。

Aurora 前端只调用 Aurora 自己目录下的后端接口。

### 6.2 PC API 边界

PC 后端只暴露：

1. 直连串口连接/断开
2. 联通测试
3. 前后左右控制
4. 底盘信息读取
5. ROS 状态/调试

PC 工具不引用 `serial_terminal.py`，不接触 COM13 终端状态，不依赖 Aurora 进程。

PC 页面文案与提示统一使用“电脑直连 STM32”。

### 6.3 A1 API 边界

A1 后端只暴露：

1. COM13 会话检查
2. `A1_TEST` 命令发送
3. 底盘联通测试
4. 前后左右控制
5. 底盘信息读取

A1 工具不提供直连 STM32 端口选择，不提供 ROS 调试。

A1 页面文案与提示统一使用“经 A1 中继”。

### 6.4 工具间关系

- 三套工具之间不互调 HTTP。
- 三套 HTML 只调用本目录 Python 后端。
- 三套 `.ps1` 只启动本目录 Python。
- 不存在公共“主页面”负责转发到另外两套工具。

## 7. 错误处理与日志规则

### 7.1 Aurora

Aurora 启动只处理视频链路、Qt 桥和 COM13 终端。

`launch.ps1` 默认安静模式，只打印：

1. 监听地址
2. Qt 桥是否启动成功
3. COM13 是否可用
4. 明确报错

Aurora 不做高频刷屏重试。失败后给出明确状态，等待用户手动刷新摄像头或重启。

摄像头错误只影响视频区，不影响 COM13 终端和联通测试区。

### 7.2 PC

PC 不带 Qt 桥、不带摄像头探测、不带 Aurora 旧日志流。

启动日志只打印：

1. Web 端口
2. 串口模块是否加载
3. ROS 环境是否可用

串口失败、底盘无响应、ROS 不可用，都在各自卡片内报错，不拖垮整页。

所有自动重试关闭，动作由用户显式触发。

### 7.3 A1

A1 不带摄像头、不带 Qt 桥。

启动日志只打印：

1. Web 端口
2. COM13 / `A1_TEST` 会话是否就绪

如果 COM13 被占用，只报一次清晰错误，不无限刷屏。

`A1_TEST` 返回异常时，前端直接显示原始失败信息，方便查板端。

### 7.4 统一规则

三套工具日志前缀分开：

- `[Aurora]`
- `[PC]`
- `[A1]`

不共享异常处理器，不做后台偷偷切换目标或端口的隐式修复。

错误提示必须绑定当前工具语义，避免出现混合提示。

## 9. 实施顺序

1. 先瘦身 `tools/aurora/`，把不应保留的底盘控制与 ROS UI/路由标记清楚。
2. 从现有直连 STM32 / ROS 逻辑提炼出 `tools/PC/` 第一版，先跑通后端和最小页面。
3. 从现有 COM13 / `A1_TEST` 中继逻辑提炼出 `tools/A1/` 第一版，先跑通后端和最小页面。
4. 三边分别补各自 `.ps1`、HTML、状态文件和日志前缀。
5. 最后删除 Aurora 中残留的底盘控制代码，避免中途功能断档。
6. 三套工具联调通过后，再补文档和启动说明。

## 10. 非目标

本次不做以下事项：

- 不抽公共 `tools/common/` 共享层。
- 不顺手重写 Aurora 视频链路。
- 不在本次拆分里顺手解决 STM32 协议设计问题。
- 不改变 A1_TEST 命令语义。
- 不引入新的前端框架。

## 11. 风险与应对

### 风险 1：迁移时把旧 Aurora 相机/Qt 问题复制到 PC/A1

应对：PC/A1 完全不接入相机与 Qt 桥启动链路，启动脚本保持极简。

### 风险 2：拆分后仍残留共享状态导致串台

应对：明确禁止共享目标切换状态、共享 PID 文件和共享连接状态文件。

### 风险 3：先删 Aurora 再建新工具，造成中途功能断档

应对：先建 `tools/PC/` 和 `tools/A1/` 最小可运行版本，最后再移除 Aurora 残留模块。

### 风险 4：用户仍在错误页面寻找错误能力

应对：三套页面标题、按钮文案、错误提示全部写死为各自职责语义，不再给用户“模式切换”错觉。
