# 文档体系重组与现状同步设计

- 日期：2026-04-27
- 范围：`README.md`、`tools/aurora/README.md`、`docs/`
- 目标：基于当前已落地代码，重组文档结构、修正文档历史包袱、统一入口信息，并为后续提交 PR 提供清晰变更边界。

## 1. 目标

本次工作不是单纯补几行说明，而是把仓库文档从“历史快照 + 局部过时描述”整理为“基于当前代码的可用知识库”。

成功标准：

1. 根 `README.md` 能准确描述仓库组成、主要构建路径、运行验证方式和文档导航。
2. `tools/aurora/README.md` 能准确覆盖 Aurora 当前真实结构、启动方式、双控制路径、调试能力和常见问题。
3. `docs/` 入口型文档之间的链接、命令、路径、端口和模块描述与当前代码保持一致。
4. 原“规划/分工/里程碑”类文档改写为“当前落地现状 + 后续方向”，不再保留明显失真的项目计划叙述。
5. 最终 PR 聚焦文档事实同步和结构重组，不混入无关代码改动。

## 2. 当前问题

从现有仓库可见，文档存在以下问题：

### 2.1 根 README 失真

当前 `README.md` 仍混有旧端口、旧文档链接格式、旧路径引用和不完整的现状描述。例如：

- Aurora 页面访问地址仍写为旧端口。
- 文档索引表中存在 Markdown 链接格式错误。
- 一些模块状态、仓库结构和协作说明没有完全对齐当前代码与脚本。

### 2.2 Aurora README 落后于当前工具形态

`tools/aurora/README.md` 已有基础内容，但仍偏旧版说明，未完全体现当前 Aurora 的实际入口、启动参数、网页端口、双控制链路、A1_TEST 调试终端和当前推荐使用方式。

### 2.3 docs 内部存在历史包袱

`docs/` 下多篇文档仍保留：

- 旧命令、旧文件名或旧引用路径
- “计划中/里程碑/成员分工”这类已不适合作为当前事实文档的表述
- 文档之间索引关系不稳定，入口层和专题层边界不够清楚

### 2.4 文档定位不统一

目前部分文档像“使用手册”，部分像“设计记录”，部分又像“项目计划草稿”，导致读者很难判断：

- 现在系统到底支持什么
- 应该从哪里开始搭环境、构建、验证
- 某篇文档是在讲现状、历史设计还是未来规划

## 3. 设计原则

1. **以当前代码为准。** 文档描述必须服从当前脚本、目录结构和已落地能力。
2. **优先修正事实，不虚构规划。** 不保留没有代码支撑的“已完成”表述。
3. **重组信息架构，不轻易大改文件名。** 为避免大面积链接断裂，本轮主要通过索引、开头定位和交叉链接重组文档结构。
4. **入口文档清晰，专题文档克制。** 对入口型文档做系统重写；对专题深文档只修导航和明显失真，避免无根据重写细节。
5. **PR 范围纯文档。** 不借机改业务逻辑、构建逻辑或前端实现。

## 5. 文档重组方案

### 5.1 根 README 方案

根 `README.md` 改成仓库总入口，强调四件事：

1. 仓库由哪些一方模块组成
2. 当前推荐构建路径是什么
3. 当前有哪些可验证运行路径
4. 深入阅读应该进入哪些文档

结构上采用：

- 项目简介
- 仓库组成与边界
- 快速开始 / 常用命令
- 构建与验证说明
- Aurora / 板端 / ROS2 简要入口
- 文档索引
- 协作说明

内容上统一对齐当前脚本和约定：

- `scripts/bootstrap.sh`
- `scripts/build_complete_evb.sh`
- `scripts/build_ros2_ws.sh`
- `scripts/build_incremental.sh`
- `tools/aurora/launch.ps1`
- Aurora 默认端口 `http://127.0.0.1:5801`

### 5.2 Aurora README 方案

`tools/aurora/README.md` 改成 Aurora 专项手册，不再只做简短说明。

建议结构：

- Aurora 定位与适用场景
- 目录结构与关键模块职责
- 安装依赖与启动方式
- 启动参数说明
- 网页 UI 与核心功能
- 双控制路径说明（直连 STM32 / 经由 A1）
- A1_TEST 串口终端与常用命令
- 常见问题与排障
- 与主仓库其余部分的关系

写法上明确：

- Aurora 负责 Windows 侧测试、预览与联调，不负责固件烧录本身
- Flask 路由、相机桥、串口终端、relay、ROS bridge 分别做什么
- 当前推荐调试路径和已知约束是什么

### 5.3 docs 信息架构方案

不大规模改文件名。本轮通过以下方式重组：

1. 在入口文档开头补上“本文定位 / 适合谁读 / 相关文档”
2. 在根 README 和关键 docs 中统一索引关系
3. 清理失效引用、旧文件名引用和错误的 Markdown 链接
4. 让 `docs/` 更像“从入门到专题”的分层知识库，而不是松散文档堆

推荐分层心智模型：

- 入门层：快速上手、环境、编译、FAQ
- 现状层：程序概览、架构设计、项目现状
- 协作层：贡献指南、后续建议
- 专题层：ROS、模型训练、雷达、模型部署、深度感知

### 5.4 项目规划文档改写方案

`docs/12_项目规划.md` 不再保留“成员分工 + Sprint 里程碑 + 计划打勾”这种已过时结构。

改写为：

- 当前项目组成
- 当前已落地能力
- 当前集成边界和未纳入默认路径的部分
- 后续方向（按主题列出）

这样既保留“路线感”，又避免把旧计划冒充现状。

## 6. 具体执行策略

### 6.1 事实来源

所有文档改写以以下来源为准：

- 当前仓库目录结构
- `CLAUDE.md` 中已整理的仓库工作说明
- 当前脚本和入口文件
- Aurora 当前 Python 入口与模板结构
- 近期提交历史里已经落地的功能

### 6.2 命令与路径统一

会系统修正：

- 旧端口
- 旧路径
- 旧命令参数
- 错误或断裂的 Markdown 链接
- 与当前脚本不一致的构建说明

### 6.3 写法统一

文档统一回答四类问题：

1. 现在这块代码负责什么
2. 应该怎么启动 / 构建 / 验证
3. 依赖哪些边界条件
4. 如果继续深入，下一篇该看什么

避免使用：

- 无证据的“已完成/已支持”
- 过时的成员分工表
- 无法从当前代码验证的路线图断言

## 7. 验证与交付

### 7.1 文档自检

至少检查以下内容：

- 关键 Markdown 链接是否合理
- 关键命令是否与当前脚本名一致
- README 与 Aurora README 中端口、路径、入口是否一致
- `docs/12_项目规划.md` 是否已完成“计划文档 → 现状文档”转换

### 7.2 低成本验证

必要时执行最小命令校验，例如：

- `git grep` / 文本搜索确认旧端口和旧引用已清理到入口层文档之外
- Python 语法检查（如果本轮顺手调整了文档中引用的 Aurora 文件名，需要对照现有文件确认）

### 7.3 PR 输出

最终 PR 聚焦三类变更：

1. 文档结构重组
2. 文档事实同步
3. 规划文档现状化重写

PR 描述应明确说明：本次不修改核心业务代码，只提升文档可信度、可导航性和与当前代码的一致性。

## 8. 风险与控制

### 风险 1：文档改写过大导致旧链接断裂

应对：本轮尽量不改文件名，通过索引和交叉链接重组信息架构。

### 风险 2：专题深文档被过度重写，反而引入新失真

应对：专题文档只修事实错误和导航关系，不凭猜测重写技术细节。

### 风险 3：PR 混入代码改动

应对：明确只编辑文档文件；创建 PR 前检查 diff 范围。

## 9. 非目标

本次不做以下事项：

- 不重命名整套 docs 文件
- 不调整构建脚本、Aurora 逻辑或 ROS2 实现
- 不补写没有代码依据的大而全设计文档
- 不清理与本次文档工作无关的仓库改动
