# 核心亮点预览

> 以下为本作品的核心技术亮点，详细内容见后续各章节。

## ① 完整工程闭环

**从数据采集到板端控制，全链路打通且可复现。**

```
数据采集 → 模型训练 → ONNX 导出 → A1 板端部署 → STM32 底盘控制 → Windows 联调验证
```

项目已建立从视频数据预处理、五分类模型训练、ONNX 导出、`.m1model` 转换、EVB 镜像构建到上板运行的标准化流程。每一步都有固定脚本、固定参数和可验证输出，不是"能跑就行"的临时方案。

## ② 方案创新：YOLO → 轻量五分类

**放弃复杂检测框，改用 MobileNetV1 五分类直接输出动作语义。**

| 对比维度 | YOLO 检测方案 | 五分类方案（当前主线） |
|---------|-------------|-------------------|
| 输出类型 | 检测框 + 类别 | 直接类别 logits |
| 后处理 | 坐标还原、阈值筛选 | 无需后处理 |
| 推理链路 | 长 | 短 |
| 动作映射 | 需额外转换 | 直接映射 |
| A1 算子兼容 | 受限 | 完全兼容 |
| 联调复杂度 | 高 | 低 |

五分类类别：`person` / `stop` / `forward` / `obstacle` / `NoTarget`，直接对应底盘控制语义（`forward` → 前进， → `stop`停止）。

## ③ 三套 Windows 联调工具

**视觉侧、控制侧、整链路三层拆分，调试路径不再混淆。**

| 工具 | 目录 | 核心职责 | 验证对象 |
|------|------|---------|---------|
| **Aurora** | `tools/aurora/` | 视频预览、拍照、COM13 终端、A1_TEST 调试 | A1 视觉与板端状态 |
| **PC** | `tools/PC/` | PC 直连 STM32 串口调试 | STM32 底盘协议 |
| **A1** | `tools/A1/` | COM13 → A1_TEST → STM32 中继控制 | A1 端到 STM32 整链路 |

三套工具不是重复功能，而是分别独立验证不同层级，确保问题不会被错误归因。

## ④ 可复现训练体系

**固定数据处理管线 + 完整评估报告，训练结果可回溯。**

- **数据管线**：视频 → 抽帧 → 裁剪 → 灰度化 → 划分 train/val/test
- **训练策略**：AdamW + 余弦退火 + Dropout + Label Smoothing + Early Stopping
- **评估输出**：`results.csv` + 训练曲线 + 混淆矩阵（val/test）+ 每类 Precision/Recall/F1

训练脚本在启动时强制校验 train/val 各类别样本完整性，避免中途因缺类失败。

## ⑤ 清晰的模块边界

**A1 负责「看」和「判」，STM32 负责「动」，Windows 负责「查」和「测」。**

```
摄像头 → A1 视觉板（采集+推理+OSD+UART）→ STM32（底盘控制）→ 小车执行
                ↑                                    ↑
          Aurora 工具                            PC / A1 工具
```

三层解耦的好处：A1 不承担复杂动作规划，STM32 不承担视觉推理，Windows 不参与固件烧录主链路。每层独立可测，问题定位不跨层。

# 1. 项目概述

## 1.1 项目背景

本项目面向 SmartSens A1 视觉板与 STM32 智能小车联动场景，搭建了一套端侧视觉识别控制系统。A1 负责图像采集、AI 推理、OSD 显示和 UART 指令输出，STM32 负责底盘动作执行，Windows 上位机工具负责相机预览、串口调试和联调验证。

当前中期主线已经从目标检测方案转为轻量五分类识别方案。这样做的原因很直接：五分类模型的输出更简单，推理链路更短，部署和联调也更稳定。

当前五分类类别为 `person`、`stop`、`forward`、`obstacle` 和 `NoTarget`。其中，前四类对应典型视觉状态，`NoTarget` 表示空场景或无有效目标，作为统一训练和推理的一部分保留。

## 1.2 项目目标

本项目不追求通用识别能力，重点是把 A1 板端演示链路做稳。具体目标如下：

1. 完成 A1 端摄像头图像采集与预览；
2. 在 A1 端运行轻量五分类模型；
3. 通过 OSD 显示识别结果、系统状态和动作提示；
4. 将视觉识别结果通过 UART 发送给 STM32；
5. 由 STM32 控制小车完成前进、停止等动作；
6. 使用 Windows 工具辅助预览、调试和联调；
7. 建立从数据处理、训练、验证、测试到部署验证的完整流程。

## 1.3 中期阶段重点

中期阶段的重点已经从方案探索转向主线收敛。项目前期尝试过 YOLO 检测式方案，但在当前部署约束和演示目标下，稳定性和匹配度都不够，因此主线调整为五分类方案。

这一阶段主要完成了以下工作：

- 明确系统结构和模块边界；
- 跑通五分类训练链路；
- 梳理板端运行、Windows 调试和 STM32 控制路径；
- 完善训练结果记录、混淆矩阵和每类指标输出；
- 准备中期报告、PPT 和演示视频所需材料。

整体上看，项目已经从功能探索走到可训练、可导出、可部署、可验证的工程阶段。

# 2. 系统总体设计

## 2.1 总体架构

本项目采用“视觉感知 + 状态显示 + 通信控制 + 运动执行 + 上位机联调”的分层结构。A1 负责图像采集、五分类推理、OSD 显示和 UART 指令输出；STM32 负责底盘电机控制；Windows 工具负责相机预览、串口调试和联调验证。

数据流比较清晰：摄像头采集图像后进入 A1 板端，经过预处理和模型推理得到五分类结果，再把分类语义映射为控制指令，通过 UART 或 A1_TEST 调试路径发送给 STM32。STM32 收到指令后驱动底盘动作，同时 A1 侧通过 OSD 显示当前状态，方便观察和排查问题。

```text
摄像头模组
  ↓
SmartSens A1 视觉板
  ├─ 图像采集
  ├─ 五分类模型推理
  ├─ 分类结果解析
  ├─ OSD 状态显示
  └─ UART 控制指令发送
        ↓
      STM32 控制板
        ↓
  小车底盘 / 电机驱动 / 执行动作

Windows 上位机工具
  ├─ Aurora 视频预览
  ├─ COM13 串口终端
  ├─ A1_TEST 调试命令
  ├─ PC 直连 STM32 调试
  └─ A1 中继控制验证
```

## 2.2 系统分层

| 层级 | 主要模块 | 功能 |
| --- | --- | --- |
| 视觉感知层 | A1 摄像头与 AI demo | 图像采集、模型推理、结果解析 |
| 显示反馈层 | OSD 图层 | 叠加状态提示、动作提示和调试信息 |
| 通信控制层 | A1 UART / A1_TEST | 将识别状态转换为控制指令 |
| 运动执行层 | STM32 与底盘 | 接收指令并控制小车运动 |
| 上位机调试层 | Aurora / PC / A1 工具 | 视频预览、串口调试、联调验证 |
| 构建部署层 | SDK 与脚本 | 编译应用并打包 EVB 镜像 |

## 2.3 当前代码结构对应关系

| 目录 | 作用 |
| --- | --- |
| `demo-rps/dataprocess_modeltrain/` | 五分类数据处理、训练、ONNX 导出与测试 |
| `data/rps_dataset/` | 视频数据、处理后数据集、训练输出与模型文件 |
| `data/A1_SDK_SC132GS/smartsens_sdk/smart_software/src/app_demo/face_detection/ssne_ai_demo/` | A1 板端推理、OSD、A1_TEST、UART 逻辑 |
| `tools/aurora/` | Windows 摄像头预览、COM13 终端和 A1 调试入口 |

| `tools/A1/` | PC 经 A1 中继控制 STM32 的调试工具 |

| `docs/` | 环境、架构、训练、烧录、提交材料说明 |

## 2.4 两条重要控制与调试路径

项目里要区分两条独立路径，不然问题很容易定位错层。

### 2.4.1 直连 STM32 路径

PC 串口 → STM32 UART。该路径主要由 `tools/PC/pc_chassis.py` 负责，适合直接看底盘运动和串口遥测，用来验证底盘协议、波特率、接线和底层控制逻辑。

### 2.4.2 经由 A1 路径

PC COM13 → A1_TEST → A1 UART0 → STM32 UART3。该路径主要由 `tools/A1/a1_relay.py` 与 `tools/A1/a1_serial.py` 负责，适合联调板端程序和底盘联动，用来确认板端识别结果能否正确转成控制语义并传给 STM32。

## 2.5 设计边界

A1 负责“看”和“判”，STM32 负责“动”，Windows 工具负责“查”和“测”。A1 不承担复杂动作规划，STM32 不承担视觉推理，Windows 侧不参与固件烧录主链路，只负责调试和验证。这个边界划分能减少耦合，也方便中期阶段定位问题。

# 3. 技术路线演进

## 3.1 早期 YOLO 检测方案

项目早期尝试过 YOLO 目标检测方案，希望通过检测框识别目标位置和类别，再映射到 OSD 显示和底盘控制。这个方案的优点是表达能力强，能同时给出类别和位置，适合通用检测任务。

但在当前场景里，它也有明显问题：

- 模型结构更复杂，板端转换和算子约束更严格；
- 输入输出后处理链路更长；
- 检测框还原、阈值筛选和类别映射增加联调复杂度；
- 上版测试效果不理想，演示稳定性不够。

所以，YOLO 更适合作为前期探索和技术储备，不适合作为中期展示主线。

## 3.2 YOLO 方案在报告中的定位

YOLO 方案需要保留，但只作为早期探索和路线迭代记录，不作为当前代码展示重点。报告里只写方案尝试、测试反馈和切换原因，不展开检测网络细节，这样不会冲淡当前五分类主线。

推荐表述如下：

> 项目前期曾尝试基于 YOLO 的检测式方案，但上版测试中稳定性和部署匹配度未达到当前演示目标，因此中期阶段将主线调整为轻量五分类模型方案。YOLO 方案作为前期探索和方案演进记录保留，不作为当前展示代码重点。

## 3.3 当前五分类模型主线

当前主线采用轻量五分类模型，把视觉状态直接划分为 `person`、`stop`、`forward`、`obstacle` 和 `NoTarget`。和 YOLO 相比，这条方案更适合当前中期演示，原因很明确：

- 输出语义直接，便于动作映射；
- 推理链路更短；
- 不需要检测框坐标还原；
- 更容易满足 A1 算子支持范围；
- 训练、评估、导出和部署流程更稳定。

这条路线的核心不是做更复杂的视觉能力，而是让识别、控制和展示都稳定。

## 3.4 版本演进脉络

项目提交历史跨度较长，当前可见提交已覆盖 300+ 次变更。整体演进不是单线推进，而是多个模块并行收敛，最后在 A1 五分类主线上形成闭环。按阶段看，可以分成几条脉络。

### 3.4.1 仓库初始化与基础设施阶段

最早阶段主要做仓库落地、SDK 导入、CI 和 gitignore 规范、基础构建入口整理。目标不是先做完整演示，而是先把工程骨架搭起来，保证后续能持续集成、能重复构建、能追踪变更。

这一阶段的特点是：

- SDK 树逐步导入并整理；
- 构建脚本和 Docker 环境开始成形；
- `.gitignore`、文档骨架和 CI 流程开始稳定；
- `ssne_ai_demo`、`demo_app` 等目录命名和位置逐步明确。

### 3.4.2 早期功能探索阶段

随后项目开始围绕图像显示、驱动适配、ROS/底盘集成和基础联通测试做探索。这个阶段的提交里可以看到：

- 图像 dump 和显示能力逐步增加；
- UART、GPIO、底盘驱动和 `libssne` 版本不断适配；
- 文档从零散记录变成可阅读的项目说明；

这个阶段的重点是先摸清 A1 能接什么、能跑什么、哪些模块会冲突，再决定主线。

### 3.4.3 构建链路与文档体系成型阶段

到 3 月中下旬，提交开始集中到“能构建、能复现、能交付”这条线上。关键变化包括：

- 完整 EVB 构建链路统一；
- Docker / build 脚本逐步稳定；

这个阶段说明项目已经从“能跑的零散代码”转向“能交的工程项目”。

### 3.4.4 Aurora 与 Windows 联调工具扩展阶段

4 月开始，Windows 工具变成项目推进的重点之一。Aurora 从单纯视频预览逐步扩展为：

- 摄像头预览；
- 拍照和训练集采样；
- COM13 串口终端；
- A1_TEST 手动调试；
- A1 视觉预览和状态查看；
- 与板端、底盘联动的测试界面。

这个阶段说明项目不再只是改板端代码，而是在一起打通“视觉输入—调试界面—串口通信—联调验证”整条链路。

### 3.4.5 视觉主线切换阶段

4 月中旬到 5 月初，视觉方案经历了几次明显迭代：

- 从早期检测式方案探索出发；
- 到 YOLOv8 head6 检测方案；
- 再到当前五分类方案收敛。

这条主线反映的变化很清楚：

- 目标从“做复杂检测”转为“做稳定识别与控制”；
- 部署约束从“通用检测网络”转为“A1 可落地算子”；
- 结果输出从“检测框 + 后处理”转为“直接类别 + 动作映射”。

也就是说，提交历史不是简单换模型，而是项目目标逐步收敛到可展示、可部署、可验收方案的过程。

### 3.4.6 训练链路完善阶段

5 月初训练脚本逐步补全，五分类数据处理、训练、验证、测试和导出形成闭环。这个阶段最重要的变化包括：

- 训练输入从视频探索转为稳定图片数据集；
- 数据集划分从按视频改为按帧；
- 训练脚本增加正则化、早停和完整报告输出；
- 导出流程开始围绕 A1 算子兼容性约束设计。

这一阶段把“有模型”变成“有可复现训练流程的模型”。

### 3.4.7 当前收敛阶段

最新提交已经把主线收敛到：

- 五分类模型；
- 训练输出和评估报告；
- A1 板端资源和运行脚本；
- Aurora / PC / A1 三套测试工具；

当前项目的重点不再是继续加功能，而是把现有链路稳定下来，把证据、报告和交付材料补齐。

# 4. 数据集与数据处理

## 4.1 数据来源

训练数据来自 `data/rps_dataset/` 下采集的视频素材。原始数据按类别组织，覆盖 `person`、`stop`、`forward`、`obstacle` 和 `NoTarget` 等状态。视频内容包含不同拍摄角度、不同距离和不同光照条件下的目标状态，用于支撑五分类模型训练。

当前项目不直接用视频训练，而是先把视频转换成裁剪后的图片数据集，再送入训练脚本。这样做的好处是数据格式更稳定，训练输入更统一，也更方便后续导出和板端部署保持一致。

## 4.2 类别设计

五分类任务的类别定义如下：

| 类别 | 含义 | 用途 |
| --- | --- | --- |
| `person` | 人物或人体目标 | 人物状态识别 |
| `stop` | 停止手势或停止状态 | 触发停止语义 |
| `forward` | 前进手势或前进状态 | 触发前进语义 |
| `obstacle` | 障碍物状态 | 用于避障或告警语义 |
| `NoTarget` | 无有效目标 | 空场景、待机或兜底状态 |

这套类别定义的目标不是扩展通用识别能力，而是让模型输出可以直接映射到项目中的动作语义。`NoTarget` 的加入也很重要，它保证空场景下系统仍然有明确输出，避免板端在无目标时产生不稳定控制。

## 4.3 数据预处理流程

当前数据处理流程已经固定为“视频 → 抽帧 → 裁剪 → 灰度化 → 分割数据集”五个步骤：

1. 从原始视频读取帧；
2. 按固定帧间隔抽样，减少连续帧冗余；
3. 对画面做中心裁剪，再截取固定区域；
4. 将裁剪结果转为灰度图并保存为图片；
5. 按类别和帧数划分为 `train / val / test`。

对应处理脚本为 `demo-rps/dataprocess_modeltrain/prepare_video_dataset.py`。该脚本默认从 `data/rps_dataset/datasets/` 读取视频，输出到 `data/rps_dataset/processed_dataset/` 或当前使用的 `data/rps_dataset/processed_dataset_regularized/`。

脚本关键参数如下：

- `--dataset_dir`：原始视频目录；
- `--output_dir`：处理后数据集目录；
- `--frame_step`：抽帧间隔，默认 `3`；
- `--train_ratio`：训练集比例，默认 `0.8`；
- `--val_ratio`：验证集比例，默认 `0.1`；
- `--crop`：固定裁剪区域，默认 `0 0 320 320`；
- `--image_ext`：输出图片格式，默认 `.png`。

## 4.4 处理后数据集结构

当前稳定使用的数据集目录为：

```text
data/rps_dataset/processed_dataset_regularized/
  train/
    person/
    stop/
    forward/
    obstacle/
    NoTarget/
  val/
    person/
    stop/
    forward/
    obstacle/
    NoTarget/
  test/
    person/
    stop/
    forward/
    obstacle/
    NoTarget/
```

这个目录结构和训练脚本输入要求一致。训练脚本会直接读取 `train`、`val`、`test` 三个 split 下的类别目录，并按目录名映射到固定标签顺序。

## 4.5 数据划分优化

早期如果按视频数量划分，容易出现某些 split 缺少某一类别的问题，尤其是样本量较少的 `NoTarget` 类。这样一来，训练脚本在启动时就可能因为某个 split 缺类而直接失败，影响演示和重复训练效率。

当前方案改为按抽样帧数量划分，而不是单纯按视频数量划分。这样后，train、val、test 更容易同时覆盖全部类别，训练稳定性更高，也更适合小样本、固定类别的工程任务。

## 4.6 当前数据状态

当前项目已经形成较稳定的数据准备结果，包括：

- 原始视频数据集；
- 处理后的帧图片数据集；
- 平衡后的训练数据；
- 训练输出权重和元数据；
- 可继续用于后续训练和导出的标准化目录。

这说明数据链路已经从“能采集”发展到“能稳定训练”，可以支撑后续模型训练、评估和部署。

# 5. 五分类模型训练设计

## 5.1 任务定义

本项目训练任务是单标签五分类图像分类。输入是单通道灰度图，输出是五个类别 logits，最后取最大值对应类别作为识别结果。这个定义和板端控制语义直接对齐，避免了目标检测方案里的检测框还原和多阶段后处理。

训练脚本中类别顺序固定为：

```python
CLASS_NAMES = ["person", "stop", "forward", "obstacle", "NoTarget"]
NUM_CLASSES = len(CLASS_NAMES)
INPUT_CHANNELS = 1
```

类别顺序在训练、导出和板端推理中必须保持一致，不然识别结果和动作映射会错位。

## 5.2 模型结构

本项目采用 MobileNetV1 作为骨干网络，输入尺寸固定为 `320×320`，分类头采用轻量卷积结构。这个结构兼顾了算力开销、导出稳定性和板端部署兼容性。

训练脚本中的模型定义如下：

```python
self.backbone = timm.create_model(
    "mobilenetv1_100",
    pretrained=False,
    num_classes=0,
    global_pool="",
    in_chans=INPUT_CHANNELS,
)
...
self.head = nn.Sequential(
    nn.Conv2d(feature_channels, head_hidden_dim, kernel_size=1, bias=False),
    nn.BatchNorm2d(head_hidden_dim),
    nn.ReLU(inplace=True),
    nn.Dropout(p=dropout),
    nn.AvgPool2d(kernel_size=2, stride=2),
    nn.AdaptiveAvgPool2d(1),
    nn.Conv2d(head_hidden_dim, NUM_CLASSES, kernel_size=1, bias=True),
)
```

这个结构的特点是：

- Backbone 采用轻量级卷积网络，不引入复杂检测头；
- 输入通道为 `1`，与板端灰度图像输入一致；
- 分类头参数规模较小，便于训练收敛和模型转换；
- 输出直接为五类 logits，不包含 Softmax 等不必要算子。

## 5.3 训练策略

训练阶段采用数据增强、正则化和早停组合策略，提升小样本场景下的泛化稳定性。训练脚本主要参数如下：

| 参数 | 默认值 | 作用 |
| --- | --- | --- |
| `batch_size` | `64` | 控制单轮训练批大小 |
| `epochs` | `30` | 最大训练轮数 |
| `lr` | `3e-4` | 学习率 |
| `weight_decay` | `1e-3` | 权重衰减 |
| `head_hidden_dim` | `128` | 分类头隐藏通道数 |
| `dropout` | `0.4` | Dropout 比例 |
| `label_smoothing` | `0.1` | 标签平滑 |
| `patience` | `5` | 早停耐心轮数 |
| `min_delta` | `0.002` | 最小有效提升阈值 |

数据增强策略主要包括：

- `Resize + RandomResizedCrop`；
- `RandomColorJitter`；
- `RandomRotation`；
- `RandomHorizontalFlip`；
- `RandomErasing`；
- `Normalize`。

验证和测试阶段只使用确定性的 `Resize + Normalize`，不引入随机增强。

## 5.4 训练实现

训练脚本 `demo-rps/dataprocess_modeltrain/train_a1_5class_classifier.py` 启动后会先检查数据集目录，再校验 `train` 与 `val` split 中每个类别是否存在样本。核心校验逻辑如下：

```python
def verify_required_classes(split_name: str, samples_by_class):
    missing = [class_name for class_name in CLASS_NAMES if not samples_by_class.get(class_name)]
    if missing:
        raise RuntimeError(f"{split_name} split missing class data: {missing}")
```

这个检查是为了避免训练中途因为类别缺失导致损失函数无法正常收敛，或者评估结果失真。

训练过程采用 AdamW 优化器和余弦退火学习率调度器：

```python
criterion = nn.CrossEntropyLoss(label_smoothing=args.label_smoothing)
optimizer = torch.optim.AdamW(model.parameters(), lr=args.lr, weight_decay=args.weight_decay)
scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=max(args.epochs, 1))
```

验证集指标优于历史最佳值时保存 `best.pt`，每轮训练都保存 `last.pt`。当验证集连续若干轮未达到最小增益阈值时，触发 early stopping：

```python
if epochs_without_improvement >= args.patience:
    print(f"Early stopping at epoch {epoch:03d} after {epochs_without_improvement} stale epochs")
    break
```

# 6. 模型评估与结果分析

## 6.1 核心评估指标

本项目的评估指标覆盖整体准确率、训练稳定性和类别均衡性三个层面。训练脚本记录的核心指标包括：

- `train_loss`；
- `val_loss`；
- `train_top1`；
- `val_top1`；
- `test_top1`；
- 每类 `precision`；
- 每类 `recall`；
- 每类 `F1`；
- 混淆矩阵。

其中，`top1` 表示模型输出最大 logit 是否对应真实类别；混淆矩阵用于看类别间误判关系；每类 F1 用于判断各类别识别是否均衡。

## 6.2 评估实现

训练脚本中通过 logits 最大值计算预测类别：

```python
def accuracy_from_logits(logits: torch.Tensor, targets: torch.Tensor):
    preds = logits.argmax(dim=1)
    return (preds == targets).float().mean().item()
```

混淆矩阵以真实类别为行、预测类别为列进行统计：

```python
def confusion_matrix(preds: torch.Tensor, targets: torch.Tensor, num_classes: int):
    matrix = torch.zeros((num_classes, num_classes), dtype=torch.int64)
    for target, pred in zip(targets.tolist(), preds.tolist()):
        matrix[target, pred] += 1
    return matrix
```

每类指标由混淆矩阵计算得到：

```python
precision = tp / (tp + fp) if tp + fp else 0.0
recall = tp / (tp + fn) if tp + fn else 0.0
f1 = 2 * precision * recall / (precision + recall) if precision + recall else 0.0
```

这套评估方式能同时反映整体性能和单类误判情况，适合用于中期报告和答辩展示。

## 6.3 结果文件生成

训练结束后，脚本统一生成 CSV 和图像报告。实现入口如下：


## 6.4 分析重点

中期阶段模型评估不只看最终准确率，还要看以下内容：

1. 训练集和验证集 loss 是否同步下降；
2. 验证集 top1 是否稳定；
3. 是否出现训练集提升快、验证集停滞的过拟合现象；
4. `forward` 与 `stop` 是否存在明显混淆；
5. `NoTarget` 是否被误判为有效目标；
6. 每类 F1 是否均衡。

其中，`forward` 和 `stop` 的混淆会直接影响底盘动作，`NoTarget` 的误判会影响空场景安全性，所以这两类需要作为后续数据补充和训练优化重点。

A1 端部署要求模型导出图里的算子落在当前支持范围内。本项目五分类模型采用轻量卷积结构，导出后主要使用以下算子：

导出阶段刻意避免下列不适合当前链路的算子：

- `Sub`；
- `Div`；
- `Softmax`；
- Transformer 结构相关算子。

导出脚本中使用固定输入形状和静态图，避免动态维度带来的转换风险：

```python
torch.onnx.export(
    export_model,
    dummy_input,
    str(output_path),
    input_names=["input"],
    output_names=["logits"],
    opset_version=args.opset,
    external_data=False,
    do_constant_folding=True,
    dynamic_axes=None,
)
```

导出后还会再做一次 Identity 节点清理，减少图里冗余节点，提升转换稳定性。

## 7.3 模型导出链路

导出脚本 `demo-rps/dataprocess_modeltrain/export_onnx.py` 采用“读取权重 → 恢复模型 → 构造静态输入 → 导出 ONNX → 清理图结构 → 保存结果”的固定流程。其关键约束如下：

- 只支持 `320×320` 输入尺寸；
- 只支持固定五分类输出；
- 通过 `metadata.json` 读取训练参数；
- 导出输出名固定为 `input` 和 `logits`。

导出过程中的模型定义如下：

```python
self.backbone = timm.create_model(
```

这个结构避免在分类头里引入不必要的算子分支，有利于板端转换。

## 8.2 语义映射设计

当前控制逻辑以稳定优先，不引入复杂动作规划。板端代码中，识别标签到动作的映射关系如下：

```cpp

    return label == "forward" ? "forward" : "stop";
}
```

即：

- `forward` 映射为前进；
- 其余类别映射为停止。

底盘速度输出对应如下：

```cpp
constexpr int16_t kForwardVelocity = 200;

int16_t vx_for_action(const std::string& action) {
    return action == "forward" ? kForwardVelocity : 0;
}
```

这个设计保持了中期阶段动作控制的简单性和可验证性，也降低了联调复杂度。

## 8.3 OSD 显示设计

OSD 用来把系统状态可视化，主要展示：

- 当前分类结果；
- 当前动作语义；
- 空闲或无目标状态；
- 调试阶段辅助信息。

这部分设计的目标不是做视觉装饰，而是让板端运行状态能被直接看见。因为当前项目同时存在摄像头画面、分类结果和底盘控制三个层次，OSD 是最直接的在线观测入口。

训练阶段的核心动作包括：

- 校验数据集目录是否存在；

- 构建训练、验证与测试数据加载器；

## 9.2 模型部署

训练完成后，模型先导出为 ONNX，再转换为 A1 板端可用格式。导出脚本固定输出名、固定输入尺寸、固定类别顺序，并禁用动态维度：

```python
torch.onnx.export(
    export_model,
    dummy_input,
    str(output_path),
    input_names=["input"],
    output_names=["logits"],
    opset_version=args.opset,
    external_data=False,
    do_constant_folding=True,
    dynamic_axes=None,
)
```

导出阶段对输入尺寸有硬性约束：

```python
if image_size != MODEL_IMAGE_SIZE:
    raise RuntimeError(
        f"this export script only supports fixed image size {MODEL_IMAGE_SIZE}, got {image_size}"
    )
```

这个限制保证导出结果和板端输入预处理完全一致，避免导出后图结构与板端实际输入不匹配。

## 9.3 固件构建

板端构建通过完整 EVB 构建脚本完成。脚本会先检查 SDK 根目录、build 脚本和发布脚本是否存在，再执行 SDK 重新打包和 `ssne_ai_demo` 重建。核心逻辑如下：

```bash
if [[ ${RUN_APP_ONLY} -eq 0 ]]; then
  echo "[build_complete_evb.sh] 先执行 SDK 发布构建，确保基础缓存存在"

```

最终产物为 `zImage.smartsens-m1-evb`，并复制到 `output/evb/<timestamp>/` 与 `output/evb/latest/`。

构建脚本还会对 SDK 内的构建控制文件做 LF 规范化，避免 Windows 环境写入 CRLF 后引发脚本异常。这一点和本仓库的 A1 SDK 集成环境直接相关。

# 11. 问题与优化方案

## 11.1 YOLO 方案效果不理想

### 问题

项目早期采用 YOLO 检测式方案进行探索，但上版测试中稳定性和部署匹配度未达到当前演示目标。主要问题包括模型结构复杂、后处理链路较长、检测框还原增加联调复杂度，以及板端转换约束更严格。

### 优化

当前主线切换为五分类模型方案。这个方案把目标直接归纳为固定语义类别，减少检测框和多阶段后处理步骤，提高了训练、导出、部署和联调的一致性。

## 11.2 小样本类别缺失

### 问题

早期按视频数量划分数据集时，小样本类别容易在某个 split 中缺失，尤其是 `NoTarget` 类。这会导致训练脚本在启动阶段因类别不完整而失败，或者让评估结果失去代表性。

### 优化

当前改为按抽样帧数量划分，并在训练前强制检查 train 与 val 的类别完整性：

```python
verify_required_classes("train", train_samples_by_class)
verify_required_classes("val", val_samples_by_class)
```

这样能提高类别覆盖率，减少训练前置失败概率。

## 11.3 过拟合风险

### 问题

在数据规模有限的情况下，模型容易出现训练集提升快、验证集提升有限的情况。这在小样本固定类别任务中很常见。

### 优化

训练脚本采用以下策略降低过拟合风险：

- 数据增强；
- `Dropout`；
- `Label smoothing`；
- `Weight decay`；
- `Early stopping`。

相关参数已固定在训练脚本中：

```python
parser.add_argument("--weight_decay", type=float, default=1e-3)
parser.add_argument("--dropout", type=float, default=0.4)
parser.add_argument("--label_smoothing", type=float, default=0.1)
parser.add_argument("--patience", type=int, default=5)
parser.add_argument("--min_delta", type=float, default=0.002)
```

## 11.4 板端算子限制

### 问题

A1 端不适合承载复杂检测头、动态图结构和部分不兼容算子。如果模型结构选得不对，导出后可能出现算子不支持或图结构不可转换的问题。

### 优化

当前模型采用 MobileNetV1 轻量结构，并在导出阶段禁用动态维度，避免引入 `Sub`、`Div`、`Softmax` 等不适合当前链路的算子。导出脚本只保留静态输入、静态输出和固定类别顺序，降低转换风险。

## 11.5 调试路径混淆

### 问题

PC 直连 STM32 与 A1 中继 STM32 两条路径容易混用。如果没有明确区分，容易把底盘协议问题误判为板端问题，或者把板端问题误判为底盘协议问题。

### 优化

当前文档和工具已经把两条路径边界划清：

- PC 直连 STM32：用于验证底盘协议和串口通信；
- A1 中继链路：用于验证板端语义到 STM32 控制的整链路传递。

这样拆开后，问题定位更清楚，联调成本也更低。

## 11.6 当前优化结论

本项目中期阶段的优化重点已经从“修复单点问题”转向“稳定工程链路”。当前优化方案都围绕数据稳定性、训练稳定性、导出稳定性和联调稳定性展开，目标是让五分类主线在中期阶段具备可复现、可部署、可验证的工程能力。
