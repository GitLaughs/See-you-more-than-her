# 2. 系统总体设计

## 2.1 总体架构

本项目采用“视觉感知 + 状态显示 + 通信控制 + 运动执行 + 上位机联调”的分层架构。系统以 SmartSens A1 视觉板为核心计算节点，负责摄像头图像采集、五分类模型推理、OSD 状态显示和 UART 指令输出；STM32 控制板负责底盘电机控制与动作执行；Windows 上位机工具负责相机预览、串口调试和整链路联调验证。

整体数据流为：摄像头采集图像后进入 A1 板端，经过预处理和模型推理输出五分类结果，再由板端将分类语义映射为控制指令，并通过 UART 或 A1_TEST 调试路径发送给 STM32。STM32 接收到指令后驱动底盘执行前进、停止等动作，同时 A1 侧通过 OSD 叠加当前状态，便于现场观察和调试。

```text
摄像头模组
  ↓
SmartSens A1 视觉板
  ├─ 图像采集
  ├─ 五分类模型推理
  ├─ 分类结果解析
  ├─ OSD 状态显示
  └─ UART 控制指令发送
        ↓
      STM32 控制板
        ↓
  小车底盘 / 电机驱动 / 执行动作

Windows 上位机工具
  ├─ Aurora 视频预览
  ├─ COM13 串口终端
  ├─ A1_TEST 调试命令
  ├─ PC 直连 STM32 调试
  └─ A1 中继控制验证
```

## 2.2 系统分层

| 层级 | 主要模块 | 功能 |
| --- | --- | --- |
| 视觉感知层 | A1 摄像头与 AI demo | 图像采集、模型推理、结果解析 |
| 显示反馈层 | OSD 图层 | 叠加状态提示、动作提示和调试信息 |
| 通信控制层 | A1 UART / A1_TEST | 将识别状态转换为控制指令 |
| 运动执行层 | STM32 与底盘 | 接收指令并控制小车运动 |
| 上位机调试层 | Aurora / PC / A1 工具 | 视频预览、串口调试、联调验证 |
| 构建部署层 | SDK 与脚本 | 编译应用并打包 EVB 镜像 |

## 2.4 两条重要控制与调试路径

项目中需要明确区分两条独立路径，否则容易把问题定位到错误层级。

### 2.4.1 直连 STM32 路径

PC 串口 → STM32 UART。该路径主要由 `tools/PC/pc_chassis.py` 负责，适合直接观察底盘运动和串口遥测，用于验证底盘协议、波特率、接线和底层控制逻辑是否正常。

### 2.4.2 经由 A1 路径

PC COM13 → A1_TEST → A1 UART0 → STM32 UART3。该路径主要由 `tools/A1/a1_relay.py` 与 `tools/A1/a1_serial.py` 负责，适合联调板端程序和底盘联动，用于验证板端识别结果是否能够正确转化为控制语义并传递到 STM32。

## 2.5 设计边界

本项目的系统边界已经明确：A1 负责“看”和“判”，STM32 负责“动”，Windows 工具负责“查”和“测”。其中，A1 侧不承担复杂动作规划，STM32 侧不承担视觉推理，Windows 侧不参与固件烧录主链路，只负责调试与验证。这样的边界划分可以减少模块耦合，也便于在中期阶段快速定位问题来源。
