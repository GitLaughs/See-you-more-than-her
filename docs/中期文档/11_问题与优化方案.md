# 11. 问题与优化方案

## 11.1 YOLO 方案效果不理想

### 问题

项目早期采用 YOLO 检测式方案进行探索，但上版测试中稳定性和部署匹配度未达到当前演示目标。其主要问题包括模型结构复杂、后处理链路较长、检测框还原增加联调复杂度，以及板端转换约束更严格。

### 优化

当前主线切换为五分类模型方案。该方案将目标直接归纳为固定语义类别，减少检测框和多阶段后处理步骤，提升了训练、导出、部署和联调的一致性。

## 11.2 小样本类别缺失

### 问题

早期按视频数量划分数据集时，小样本类别容易在某个 split 中缺失，尤其是 `NoTarget` 类。这会导致训练脚本在启动阶段因类别不完整而失败，或者让评估结果失去代表性。

### 优化

当前改为按抽样帧数量划分，并在训练前强制检查 train 与 val 的类别完整性：

```python
verify_required_classes("train", train_samples_by_class)
verify_required_classes("val", val_samples_by_class)
```

该方式能够提高类别覆盖率，减少训练前置失败概率。

## 11.3 过拟合风险

### 问题

在数据规模有限的情况下，模型容易出现训练集提升快、验证集提升有限的情况。该问题在小样本固定类别任务中较为常见。

### 优化

训练脚本采用以下策略降低过拟合风险：

- 数据增强；
- `Dropout`；
- `Label smoothing`；
- `Weight decay`；
- `Early stopping`。

相关参数已固定在训练脚本中：

```python
parser.add_argument("--weight_decay", type=float, default=1e-3)
parser.add_argument("--dropout", type=float, default=0.4)
parser.add_argument("--label_smoothing", type=float, default=0.1)
parser.add_argument("--patience", type=int, default=5)
parser.add_argument("--min_delta", type=float, default=0.002)
```

## 11.4 板端算子限制

### 问题

A1 端不适合承载复杂检测头、动态图结构和部分不兼容算子。若模型结构选择不当，导出后可能出现算子不支持或图结构不可转换问题。

### 优化

当前模型采用 MobileNetV1 轻量结构，并在导出阶段禁用动态维度，避免引入 `Sub`、`Div`、`Softmax` 等不适合当前链路的算子。导出脚本只保留静态输入、静态输出和固定类别顺序，降低转换风险。

## 11.5 调试路径混淆

### 问题

PC 直连 STM32 与 A1 中继 STM32 两条路径容易混用。如果没有明确区分，容易把底盘协议问题误判为板端问题，或者把板端问题误判为底盘协议问题。

### 优化

当前文档和工具已经明确两条路径的边界：

- PC 直连 STM32：用于验证底盘协议与串口通信；
- A1 中继链路：用于验证板端语义到 STM32 控制的整链路传递。

通过链路拆分，问题定位粒度更清晰，联调成本更低。

## 11.6 当前优化结论

本项目中期阶段的优化重点已经从“修复单点问题”转向“稳定工程链路”。当前优化方案均围绕数据稳定性、训练稳定性、导出稳定性和联调稳定性展开，目标是保证五分类主线在中期阶段具备可复现、可部署、可验证的工程能力。
