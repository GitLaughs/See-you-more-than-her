# 10. 测试方案

## 10.1 训练侧测试

训练侧测试关注数据完整性、训练流程可运行性和导出结果正确性。测试项与预期结果如下：

| 测试项 | 预期结果 |
| --- | --- |
| 数据集完整性检查 | train / val 各类别均存在样本 |
| 训练脚本运行 | 能完成训练并输出权重 |
| 验证集评估 | 能输出 `val_loss` 与 `val_top1` |
| 测试集评估 | 能输出 `test_top1` 和混淆矩阵 |
| 报告导出 | 能生成 CSV 和 PNG 图 |
| ONNX 导出 | 导出模型不含不支持算子 |

训练脚本在启动阶段直接执行类别完整性检查：

```python
verify_required_classes("train", train_samples_by_class)
verify_required_classes("val", val_samples_by_class)
```

该检查属于训练前置条件，不满足时应立即停止，而不是继续运行到训练中后段再暴露问题。

## 10.2 板端测试

板端测试用于确认模型导入、输入预处理、类别顺序、OSD 显示和串口输出是否一致。测试项与预期结果如下：

| 测试项 | 预期结果 |
| --- | --- |
| 模型加载 | 板端能找到并加载模型 |
| 输入预处理 | 输入尺寸和通道与训练一致 |
| 分类输出 | 输出类别顺序与训练一致 |
| OSD 显示 | 能显示当前状态 |
| 串口输出 | 能根据识别状态生成控制语义 |

板端代码中的类别定义固定为：

```cpp
constexpr const char* kLabels[kClassCount] = {"person", "stop", "forward", "obstacle", "NoTarget"};
```

该常量必须与训练脚本中的 `CLASS_NAMES` 完全一致，否则会出现识别结果与控制语义不一致的问题。

## 10.3 联调测试

联调测试用于确认 Aurora、PC 与 A1 三套工具分别覆盖的链路都能正常工作。测试项与预期结果如下：

| 测试项 | 预期结果 |
| --- | --- |
| Aurora 预览 | Windows 端能看到视频画面 |
| COM13 终端 | 能访问 A1_TEST 调试命令 |
| PC 直连 STM32 | STM32 能响应 PC 指令 |
| A1 中继 STM32 | STM32 能响应 A1 中转指令 |

联调过程中应优先区分“板端识别异常”和“底盘控制异常”。如果 PC 直连 STM32 正常，而 A1 中继链路异常，问题应定位到 A1_TEST、板端语义映射或中继路径；如果 PC 直连本身异常，则问题应优先定位到底盘协议、串口或 STM32 侧。

## 10.4 测试结论

当前测试方案已经覆盖训练侧、板端侧和联调侧三个层级，能够支撑中期阶段的功能验证与问题定位。该测试框架的重点不是追求复杂用例覆盖，而是确保五分类主线在训练、导出、部署和运行四个环节均能稳定闭环。
