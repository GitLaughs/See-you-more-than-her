# 中期技术报告大纲（五分类模型主线版）

## 摘要

本项目面向 SmartSens A1 视觉板与 STM32 智能小车联动场景，设计并实现一套基于端侧视觉识别的交互控制系统。系统以 A1 视觉板作为图像采集、AI 推理、OSD 显示和 UART 指令输出核心，以 STM32 控制板作为底盘运动执行核心，通过 Windows 上位机工具完成预览、串口调试和联调验证。

项目早期曾尝试 YOLO 检测模型方案，用于探索检测式视觉交互路径。但上版测试效果未达到当前演示和部署要求，因此中期阶段将主线调整为轻量五分类模型方案。当前主线更加贴合 A1 板端算子支持、实时推理和动作映射需求。

**关键词：** SmartSens A1；五分类模型；MobileNetV1；嵌入式视觉；OSD；UART；STM32；智能小车

## 1. 项目概述

### 1.1 项目背景

- 边缘 AI 与嵌入式视觉在智能小车、机器人和教学实验平台中的应用价值。
- A1 视觉板具备摄像头输入、AI 推理、OSD 显示和外设通信能力。
- STM32 控制板负责底层运动控制，适合与 A1 形成“视觉感知 + 运动执行”分层架构。

### 1.2 项目目标

1. 实现 A1 端摄像头图像采集与预览；
2. 在 A1 端运行轻量级 AI 模型，完成视觉状态识别；
3. 使用 OSD 显示识别结果、系统状态和动作提示；
4. 通过 UART 将视觉识别结果发送给 STM32；
5. STM32 根据指令控制小车完成前进、停止等动作；
6. 使用 Windows 工具辅助摄像头预览、串口调试和联调验证；
7. 形成可训练、可导出、可部署、可验证的完整工程链路。

### 1.3 中期阶段重点

- 明确系统总体结构和模块边界；
- 完成五分类模型训练链路；
- 梳理 YOLO 方案测试不理想后的技术路线调整；
- 整理 A1 板端运行、Windows 调试和 STM32 控制路径；
- 为中期报告、PPT 和演示视频提供技术材料。

## 3. 技术路线演进

### 3.1 早期 YOLO 检测方案

项目早期曾尝试基于 YOLO 的目标检测方案，希望通过检测框识别目标位置和类别，再将检测结果映射到 OSD 显示与底盘控制。

该方案的优势是表达能力强，能够输出目标类别和位置，适合目标检测类任务。但在当前项目场景中，YOLO 方案也带来以下问题：

- 模型结构更复杂，板端转换和算子约束更严格；
- 输入输出后处理链路较长；
- 检测框还原、阈值筛选和类别映射增加联调复杂度；
- 上版测试效果不理想，演示稳定性不足。

### 3.2 YOLO 方案在报告中的定位

YOLO 方案作为项目早期探索和技术路线迭代记录写入报告，但不作为当前代码展示重点。报告中只说明方案尝试、测试反馈和切换原因，不展开代码实现细节。

推荐表述：

> 项目前期曾尝试基于 YOLO 的检测式方案，但上版测试中稳定性和部署匹配度未达到当前演示目标，因此中期阶段将主线调整为轻量五分类模型方案。YOLO 方案作为前期探索和方案演进记录保留，不作为当前展示代码重点。

### 3.3 当前五分类模型主线

当前主线采用轻量五分类模型，将视觉状态直接划分为：

```text
person / stop / forward / obstacle / NoTarget
```

相比 YOLO 检测方案，五分类方案更适合当前中期演示：

- 输出语义直接，便于映射动作；
- 推理链路短，部署更轻量；
- 不需要检测框坐标还原；
- 更容易满足 A1 算子支持范围；
- 训练、评估、导出和部署流程更稳定。

## 4. 数据集与数据处理

### 4.1 数据来源

训练数据来自 `data/rps_dataset/` 下采集的视频素材。视频按类别组织，覆盖人、停止、前进、障碍物和无目标等状态。

### 4.2 类别设计

| 类别 | 含义 | 用途 |
| --- | --- | --- |
| `person` | 人物或人体目标 | 人物状态识别 |
| `stop` | 停止手势或停止状态 | 触发停止语义 |
| `forward` | 前进手势或前进状态 | 触发前进语义 |
| `obstacle` | 障碍物状态 | 用于避障或告警语义 |
| `NoTarget` | 无有效目标 | 空场景、待机或兜底状态 |

### 4.3 数据预处理流程

1. 从原始视频读取帧；
2. 按固定帧间隔抽样，减少连续帧冗余；
3. 将抽样帧保存为图片；
4. 按类别和帧数划分 `train / val / test`；
5. 训练脚本读取处理后的图片数据集。

### 4.4 处理后数据集结构

```text
data/rps_dataset/processed_dataset_regularized/
  train/
    person/
    stop/
    forward/
    obstacle/
    NoTarget/
  val/
    person/
    stop/
    forward/
    obstacle/
    NoTarget/
  test/
    person/
    stop/
    forward/
    obstacle/
    NoTarget/
```

### 4.5 数据划分优化

早期按视频数量划分时，小样本类别容易在训练集或验证集中缺失，尤其 `NoTarget` 类更明显。当前改为按抽样帧数量划分，减少类别缺失风险，提升训练稳定性。

## 5. 五分类模型训练设计

### 5.1 任务定义

当前任务是单标签五分类图像分类。输入一张灰度图，输出五个类别 logits，最终取最大 logit 对应类别作为识别结果。

### 5.2 模型结构

- Backbone：MobileNetV1；
- 输入尺寸：`320×320`；
- 输入通道：`1`；
- 输出类别数：`5`；
- 分类头：轻量卷积头 + 池化 + Flatten。

### 5.3 训练策略

| 策略 | 作用 |
| --- | --- |
| 随机裁剪 | 提升尺度与位置鲁棒性 |
| 颜色扰动 | 缓解光照变化影响 |
| 随机旋转 | 提升姿态变化适应性 |
| 随机水平翻转 | 扩充视角变化 |
| RandomErasing | 模拟局部遮挡 |
| Dropout | 降低过拟合 |
| Label smoothing | 缓解模型过度自信 |
| Weight decay | 控制参数复杂度 |
| Early stopping | 防止无效训练轮次继续过拟合 |

### 5.4 推荐训练命令

```bash
python demo-rps/dataprocess_modeltrain/train_a1_5class_classifier.py \
  --dataset_dir data/rps_dataset/processed_dataset_regularized \
  --output_dir data/rps_dataset/training_output_regularized \
  --epochs 30 \
  --batch_size 64 \
  --lr 3e-4 \
  --weight_decay 1e-3 \
  --label_smoothing 0.1 \
  --dropout 0.4 \
  --head_hidden_dim 128 \
  --patience 5 \
  --min_delta 0.002
```

### 5.5 训练输出

| 输出文件 | 说明 |
| --- | --- |
| `best.pt` | 验证集最优模型权重 |
| `last.pt` | 最后一轮模型权重 |
| `metadata.json` | 类别、输入尺寸、训练参数等元数据 |
| `results.csv` | 每轮 loss 和 top1 指标 |
| `accuracy_loss_curve.png` | 训练 / 验证曲线 |
| `confusion_matrix_val.png` | 验证集混淆矩阵 |
| `confusion_matrix_test.png` | 测试集混淆矩阵 |
| `per_class_metrics.csv` | 每类 precision / recall / F1 / support |

## 7. A1 端部署兼容性

### 7.1 算子支持约束

A1 端部署要求模型算子落在支持范围内。当前五分类模型主要使用：

- `Conv`
- `BatchNormalization`
- `Relu`
- `AveragePool`
- `GlobalAveragePool`
- `Flatten`

### 7.2 避免使用的算子

当前模型导出图不依赖以下不适合 A1 当前部署链路的算子：

- `Sub`
- `Div`
- `Softmax`
- Transformer 结构相关算子

### 7.3 部署流程

```text
  ↓
best.pt
  ↓
ONNX 导出
  ↓
best.onnx
  ↓
A1 模型转换
  ↓
.m1model
  ↓
放入 ssne_ai_demo/app_assets/models/
  ↓
完整 EVB 镜像构建
  ↓
上板验证
```

## 8. 板端运行与联调设计

### 8.1 A1 板端运行逻辑

A1 板端 demo 负责：

1. 采集摄像头图像；
2. 对图像进行模型输入预处理；
3. 调用五分类模型推理；
4. 解析输出类别；
5. 更新 OSD 显示；
6. 生成底盘控制语义；
7. 通过 UART 或 A1_TEST 路径参与联调。

### 8.2 OSD 显示设计

OSD 用于将系统状态可视化，主要展示：

- 当前分类结果；
- 当前动作语义；
- 空闲或无目标状态；
- 调试阶段辅助信息。

### 8.3 底盘控制映射

当前控制逻辑以简单稳定为优先：

| 识别语义 | 控制语义 |
| --- | --- |
| `forward` | 前进 |
| `stop` | 停止 |
| `person` | 停止或提示 |
| `obstacle` | 停止或避障提示 |
| `NoTarget` | 停止 / 待机 |

### 8.4 Windows 测试工具与联调入口

Windows 侧工具是项目推进的重要组成部分，负责把板端推理、相机预览、串口调试和底盘联动拆开验证，避免把不同链路混在一起排查。当前工具分成三套独立入口：

#### 8.4.1 Aurora 工具

| 项目 | 说明 |
| --- | --- |
| 目录 | `tools/aurora/` |
| 默认端口 | `6201` |
| 核心职责 | 视频预览、训练集拍照、COM13 终端、A1_TEST 手动测试 |
| 主要能力 | 相机画面查看、原图/训练裁剪图保存、`rps_snapshot` 状态查看、`debug_status` / `link_test` 等联通测试 |
| 适用场景 | 板端程序联调、视觉画面检查、A1_TEST 调试 |

Aurora 主要服务于“看见板端在做什么”，适合确认摄像头画面、OSD 显示、A1_TEST 回包和分类快照结果是否正常。

#### 8.4.2 PC 工具

| 项目 | 说明 |
| --- | --- |
| 目录 | `tools/PC/` |
| 默认端口 | `6202` |
| 核心职责 | 电脑直连 STM32 调试 |
| 主要能力 | 串口控制、遥测查看、TX/RX 通信帧查看、前后左右/停止控制 |
| 适用场景 | STM32 底盘联通测试、串口协议验证、底层控制排查 |

PC 工具不经过 A1 中继，适合单独验证 STM32 自身是否能正确响应控制帧，也适合确认串口协议、波特率和接线是否正常。

#### 8.4.3 A1 工具

| 项目 | 说明 |
| --- | --- |
| 目录 | `tools/A1/` |
| 默认端口 | `6203` |
| 核心职责 | COM13 → A1_TEST → STM32 中继控制 |
| 主要能力 | A1_TEST 会话检查、手动命令发送、经 A1 中继做底盘联通测试 |
| 适用场景 | 验证 A1 板端到 STM32 的整链路控制、确认板端语义是否能到达底盘 |

A1 工具用于验证“板端识别结果能否真正传到 STM32”，是板端算法与底盘联动之间的关键桥梁。

#### 8.4.4 三套工具的分工关系

| 工具 | 验证对象 | 关注点 |
| --- | --- | --- |
| Aurora | A1 视觉与板端状态 | 预览、拍照、A1_TEST、分类快照 |
| PC | STM32 直连链路 | 串口控制、协议、遥测 |
| A1 | A1 中继链路 | 板端到底盘的端到端控制 |

调试时需要明确两条路径：

```text
PC 直连 STM32：PC 串口 → STM32 UART
A1 中继路径：PC COM13 → A1_TEST → A1 UART → STM32 UART
```

这三套工具不是重复功能，而是分别覆盖“视觉侧”“控制侧”“整链路”三个层次。

## 11. 问题与优化方案

### 11.1 YOLO 方案效果不理想

**问题：** 上版 YOLO 检测方案测试效果不理想，部署链路和演示稳定性不足。

**优化：** 将主线切换为五分类模型，减少后处理复杂度，提高部署和控制映射稳定性。

### 11.2 小样本类别缺失

**问题：** 按视频数量划分时，小样本类别可能在某些 split 中缺失。

**优化：** 改为按帧数划分，并在训练前强制检查 train/val 类别完整性。

### 11.3 过拟合风险

**问题：** 数据规模有限时，模型可能训练集表现好、验证集提升有限。

**优化：** 使用数据增强、Dropout、Label smoothing、Weight decay 和 Early stopping。

### 11.4 板端算子限制

**问题：** A1 不支持部分复杂算子和 Transformer 架构。

**优化：** 选择 MobileNetV1 轻量结构，避免 `Sub`、`Div`、`Softmax` 等不适合部署的算子。

### 11.5 调试路径混淆

**问题：** PC 直连 STM32 与 A1 中继 STM32 容易混用。

**优化：** 文档和工具中明确两条链路，分别测试、分别记录。
