# 12. 分工协作与项目管理

## 12.1 协作方式

本项目采用 Git 分支协作和任务拆分方式推进，不同模块分别在训练、板端、工具、文档和构建几个方向并行完成。该方式适合当前工程结构，因为项目包含数据集准备、模型训练、板端集成、Windows 工具和完整镜像打包多个层次，任何单点推进都难以覆盖全部链路。

## 12.2 任务管理方式

项目推进过程中，任务管理主要依赖以下方式：

- 使用 Git 提交历史记录关键技术决策；
- 使用 issue 跟踪验证、功能、优化和文档类任务；
- 使用阶段性文档沉淀训练、部署和联调结果；
- 使用工具链和脚本固定关键流程，减少重复操作成本。

这种管理方式的目标不是增加流程复杂度，而是保证每个阶段的结果都可回溯、可验证、可复现。

## 12.3 提交历史反映的阶段演进

项目提交历史已经清晰体现出技术路线演进过程：

- 从 YOLO 检测探索到五分类主线收敛；
- 从训练脚本到导出脚本，再到板端资源和运行脚本；
- 从单一预览工具到 Aurora、PC、A1 三套工具拆分。

这些提交记录不是单纯的代码修改记录，而是项目阶段性成果的工程证据。中期报告可将其作为项目持续迭代、逐步收敛的重要依据。

## 12.4 当前项目状态

截至中期阶段，本项目已经完成以下工作：

- 五分类数据集已整理完成；
- 数据预处理流程已固定；
- 训练、验证、测试和导出流程已闭环；
- A1 板端主线已收敛到五分类推理；
- Windows 联调工具已拆分为 Aurora、PC、A1 三套入口；
- 构建与运行流程已形成标准链路。

## 12.5 阶段性结论

项目当前已经从探索型实现转入工程型收敛阶段。后续工作重点不再是继续扩展无边界功能，而是围绕模型稳定性、板端部署一致性、联调可用性和报告材料完整性继续完善。
