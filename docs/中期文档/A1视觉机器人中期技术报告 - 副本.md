# 1. 项目概述

## 1.1 项目背景

本项目面向 SmartSens A1 视觉板与 STM32 智能小车联动场景，构建了一套基于端侧视觉识别的交互控制系统。系统以 A1 视觉板作为图像采集、AI 推理、OSD 显示和 UART 指令输出核心，以 STM32 控制板作为底盘运动执行核心，并通过 Windows 上位机工具完成相机预览、串口调试和联调验证，形成“视觉感知—状态显示—控制执行”一体化链路。

与传统依赖目标检测框的方案相比，当前项目中期主线调整为轻量级五分类识别方案，更适合 A1 板端实时部署与演示验证。五分类方案直接输出状态语义，不需要额外进行检测框坐标还原和复杂后处理，能够缩短推理链路、降低部署难度，并提升调试与联动控制的稳定性。

当前项目的五分类类别定义为 `person`、`stop`、`forward`、`obstacle` 和 `NoTarget`。其中，`person`、`stop`、`forward`、`obstacle` 用于表达项目中的典型视觉状态，`NoTarget` 作为空场景或无有效目标状态参与统一训练与推理。该类别体系既满足中期演示对动作语义的表达需求，也为后续板端控制映射提供了清晰、稳定的输入基础。

## 1.2 项目目标

本项目的目标不是追求通用大模型效果，而是围绕 A1 板端演示链路，构建一个稳定、轻量、可部署、可调试的视觉识别与控制系统。具体目标如下：

1. 实现 A1 端摄像头图像采集与预览；
2. 在 A1 端运行轻量级五分类模型，完成视觉状态识别；
3. 通过 OSD 叠加显示识别结果、系统状态和动作提示；
4. 通过 UART 将视觉识别结果发送给 STM32 控制板；
5. 由 STM32 根据指令控制小车完成前进、停止等动作；
6. 使用 Windows 侧工具辅助摄像头预览、串口调试和联调验证；
7. 形成从数据处理、训练、验证、测试、导出到部署验证的完整工程闭环。

## 1.3 中期阶段重点

中期阶段的工作重点已经从“方案探索”转向“主线收敛”。项目早期曾尝试基于 YOLO 的检测式方案，用于探索目标检测式视觉交互路径；但在当前部署约束和演示目标下，该方案的稳定性与匹配度不足，因此中期阶段将主线调整为轻量五分类模型方案。

围绕这一调整，中期阶段重点完成了以下几项工作：

- 明确系统总体结构和模块边界；
- 完成五分类模型训练链路；
- 梳理板端运行、Windows 调试和 STM32 控制路径；
- 完善训练结果记录、混淆矩阵和每类指标输出；
- 为中期报告、PPT 和演示视频准备可复现、可展示的技术材料。

这一阶段的成果表明，项目已经从单纯的功能探索，收敛到可训练、可导出、可部署、可验证的工程实现阶段。

# 2. 系统总体设计

## 2.1 总体架构

本项目采用“视觉感知 + 状态显示 + 通信控制 + 运动执行 + 上位机联调”的分层架构。系统以 SmartSens A1 视觉板为核心计算节点，负责摄像头图像采集、五分类模型推理、OSD 状态显示和 UART 指令输出；STM32 控制板负责底盘电机控制与动作执行；Windows 上位机工具负责相机预览、串口调试和整链路联调验证。

整体数据流为：摄像头采集图像后进入 A1 板端，经过预处理和模型推理输出五分类结果，再由板端将分类语义映射为控制指令，并通过 UART 或 A1_TEST 调试路径发送给 STM32。STM32 接收到指令后驱动底盘执行前进、停止等动作，同时 A1 侧通过 OSD 叠加当前状态，便于现场观察和调试。

```text
摄像头模组
  ↓
SmartSens A1 视觉板
  ├─ 图像采集
  ├─ 五分类模型推理
  ├─ 分类结果解析
  ├─ OSD 状态显示
  └─ UART 控制指令发送
        ↓
      STM32 控制板
        ↓
  小车底盘 / 电机驱动 / 执行动作

Windows 上位机工具
  ├─ Aurora 视频预览
  ├─ COM13 串口终端
  ├─ A1_TEST 调试命令
  ├─ PC 直连 STM32 调试
  └─ A1 中继控制验证
```

## 2.2 系统分层

| 层级 | 主要模块 | 功能 |
| --- | --- | --- |
| 视觉感知层 | A1 摄像头与 AI demo | 图像采集、模型推理、结果解析 |
| 显示反馈层 | OSD 图层 | 叠加状态提示、动作提示和调试信息 |
| 通信控制层 | A1 UART / A1_TEST | 将识别状态转换为控制指令 |
| 运动执行层 | STM32 与底盘 | 接收指令并控制小车运动 |
| 上位机调试层 | Aurora / PC / A1 工具 | 视频预览、串口调试、联调验证 |
| 构建部署层 | SDK 与脚本 | 编译应用并打包 EVB 镜像 |

| 目录 | 作用 |

| `demo-rps/dataprocess_modeltrain/` | 五分类数据处理、训练、ONNX 导出与测试 |

## 2.4 两条重要控制与调试路径

项目中需要明确区分两条独立路径，否则容易把问题定位到错误层级。

### 2.4.1 直连 STM32 路径

PC 串口 → STM32 UART。该路径主要由 `tools/PC/pc_chassis.py` 负责，适合直接观察底盘运动和串口遥测，用于验证底盘协议、波特率、接线和底层控制逻辑是否正常。

### 2.4.2 经由 A1 路径

PC COM13 → A1_TEST → A1 UART0 → STM32 UART3。该路径主要由 `tools/A1/a1_relay.py` 与 `tools/A1/a1_serial.py` 负责，适合联调板端程序和底盘联动，用于验证板端识别结果是否能够正确转化为控制语义并传递到 STM32。

## 2.5 设计边界

本项目的系统边界已经明确：A1 负责“看”和“判”，STM32 负责“动”，Windows 工具负责“查”和“测”。其中，A1 侧不承担复杂动作规划，STM32 侧不承担视觉推理，Windows 侧不参与固件烧录主链路，只负责调试与验证。这样的边界划分可以减少模块耦合，也便于在中期阶段快速定位问题来源。

# 3. 技术路线演进

## 3.1 早期 YOLO 检测方案

项目早期曾尝试基于 YOLO 的目标检测方案，希望通过检测框识别目标位置和类别，再将检测结果映射到 OSD 显示与底盘控制。该方案的优势是表达能力强，能够同时输出目标类别和位置，适合通用目标检测类任务。

但在当前项目场景中，YOLO 方案也带来明显问题：

- 模型结构更复杂，板端转换和算子约束更严格；
- 输入输出后处理链路较长；
- 检测框还原、阈值筛选和类别映射增加联调复杂度；
- 上版测试效果不理想，演示稳定性不足。

因此，YOLO 方案更适合作为前期探索和技术储备，不适合作为中期展示主线。

## 3.2 YOLO 方案在报告中的定位

YOLO 方案应写入报告，但只作为项目早期探索和路线迭代记录，不作为当前代码展示重点。报告中建议只说明方案尝试、测试反馈和切换原因，不展开具体检测网络的代码细节，以免冲淡当前五分类主线。

推荐表述如下：

> 项目前期曾尝试基于 YOLO 的检测式方案，但上版测试中稳定性和部署匹配度未达到当前演示目标，因此中期阶段将主线调整为轻量五分类模型方案。YOLO 方案作为前期探索和方案演进记录保留，不作为当前展示代码重点。

## 3.3 当前五分类模型主线

当前主线采用轻量五分类模型，将视觉状态直接划分为 `person`、`stop`、`forward`、`obstacle` 和 `NoTarget`。相比 YOLO 检测方案，五分类方案更适合当前中期演示，主要体现在以下几方面：

- 输出语义直接，便于映射动作；
- 推理链路短，部署更轻量；
- 不需要检测框坐标还原；
- 更容易满足 A1 算子支持范围；
- 训练、评估、导出和部署流程更稳定。

这条技术路线的核心不是追求更复杂的视觉能力，而是围绕“能稳定识别、能稳定控制、能稳定展示”三个目标收敛。

# 4. 数据集与数据处理

## 4.1 数据来源

训练数据来自 `data/rps_dataset/` 下采集的视频素材。原始数据按类别组织，覆盖 `person`、`stop`、`forward`、`obstacle` 和 `NoTarget` 等状态。视频内容包含不同拍摄角度、不同距离和不同光照条件下的目标状态，用于支撑五分类模型训练。

从工程角度看，当前项目并不直接使用视频进行训练，而是先将视频统一转换为裁剪后的图片数据集，再送入训练脚本。这样做的好处是数据格式更稳定、训练输入更统一，也更方便后续导出和板端部署时保持一致。

## 4.2 类别设计

五分类任务的类别定义如下：

| 类别 | 含义 | 用途 |
| --- | --- | --- |
| `person` | 人物或人体目标 | 人物状态识别 |
| `stop` | 停止手势或停止状态 | 触发停止语义 |
| `forward` | 前进手势或前进状态 | 触发前进语义 |
| `obstacle` | 障碍物状态 | 用于避障或告警语义 |
| `NoTarget` | 无有效目标 | 空场景、待机或兜底状态 |

这套类别定义的核心目标不是扩展通用识别能力，而是让模型输出能够直接映射到项目中的动作语义。`NoTarget` 的加入也很重要，它保证了空场景下系统仍然有明确输出，避免板端在无目标时产生不稳定控制。

## 4.3 数据预处理流程

当前数据处理流程已经固定为“视频 → 抽帧 → 裁剪 → 灰度化 → 分割数据集”五个步骤：

1. 从原始视频读取帧；
2. 按固定帧间隔抽样，减少连续帧冗余；
3. 对画面做中心裁剪，再截取固定区域；
4. 将裁剪结果转为灰度图并保存为图片；
5. 按类别和帧数划分为 `train / val / test`。

对应处理脚本为 `demo-rps/dataprocess_modeltrain/prepare_video_dataset.py`。该脚本默认从 `data/rps_dataset/datasets/` 读取视频，输出到 `data/rps_dataset/processed_dataset/` 或当前使用的 `data/rps_dataset/processed_dataset_regularized/`。

脚本关键参数如下：

- `--dataset_dir`：原始视频目录；
- `--output_dir`：处理后数据集目录；
- `--frame_step`：抽帧间隔，默认 `3`；
- `--train_ratio`：训练集比例，默认 `0.8`；
- `--val_ratio`：验证集比例，默认 `0.1`；
- `--crop`：固定裁剪区域，默认 `0 0 320 320`；
- `--image_ext`：输出图片格式，默认 `.png`。

当前稳定使用的数据集目录为：

## 4.5 数据划分优化

早期若按视频数量划分，容易出现某些 split 中缺少某一类别的问题，尤其是样本量较少的 `NoTarget` 类。这样一来，训练脚本在启动时就可能因为某个 split 缺类而直接失败，影响中期演示和反复训练效率。

当前方案改为按抽样帧数量划分，而不是单纯按视频数量划分。这样处理后，train、val、test 更容易同时覆盖全部类别，训练稳定性更高，也更适合小样本、固定类别的工程任务。

# 5. 五分类模型训练设计

## 5.1 任务定义

本项目训练任务为单标签五分类图像分类。输入为单通道灰度图，输出为五个类别 logits，最终取最大值对应类别作为识别结果。该任务定义与板端控制语义直接对齐，避免了目标检测方案中的检测框还原与多阶段后处理。

训练脚本中类别顺序固定为：

```python
CLASS_NAMES = ["person", "stop", "forward", "obstacle", "NoTarget"]
NUM_CLASSES = len(CLASS_NAMES)
INPUT_CHANNELS = 1
```

类别顺序在训练、导出和板端推理中必须保持一致，否则会导致识别结果与动作映射错位。

## 5.2 模型结构

本项目采用 MobileNetV1 作为骨干网络，输入尺寸固定为 `320×320`，分类头采用轻量卷积结构。该结构兼顾了算力开销、导出稳定性和板端部署兼容性。

训练脚本中的模型定义如下：

```python
self.backbone = timm.create_model(
    "mobilenetv1_100",
    pretrained=False,
    num_classes=0,
    global_pool="",
    in_chans=INPUT_CHANNELS,
)
...
self.head = nn.Sequential(
    nn.Conv2d(feature_channels, head_hidden_dim, kernel_size=1, bias=False),
    nn.BatchNorm2d(head_hidden_dim),
    nn.ReLU(inplace=True),
    nn.Dropout(p=dropout),
    nn.AvgPool2d(kernel_size=2, stride=2),
    nn.AdaptiveAvgPool2d(1),
    nn.Conv2d(head_hidden_dim, NUM_CLASSES, kernel_size=1, bias=True),
)
```

该结构的核心特点如下：

- Backbone 采用轻量级卷积网络，避免引入复杂检测头；
- 输入通道为 `1`，与板端灰度图像输入一致；
- 分类头参数规模较小，便于训练收敛和模型转换；
- 输出直接为五类 logits，不包含 Softmax 等不必要算子。

## 5.3 训练策略

训练阶段采用数据增强、正则化和早停组合策略，以提升小样本场景下的泛化稳定性。训练脚本主要参数如下：

| 参数 | 默认值 | 作用 |
| --- | --- | --- |
| `batch_size` | `64` | 控制单轮训练批大小 |
| `epochs` | `30` | 最大训练轮数 |
| `lr` | `3e-4` | 学习率 |
| `weight_decay` | `1e-3` | 权重衰减 |
| `head_hidden_dim` | `128` | 分类头隐藏通道数 |
| `dropout` | `0.4` | Dropout 比例 |
| `label_smoothing` | `0.1` | 标签平滑 |
| `patience` | `5` | 早停耐心轮数 |
| `min_delta` | `0.002` | 最小有效提升阈值 |

数据增强策略主要包括：

- `Resize + RandomResizedCrop`；
- `RandomColorJitter`；
- `RandomRotation`；
- `RandomHorizontalFlip`；
- `RandomErasing`；
- `Normalize`。

验证和测试阶段仅使用确定性的 `Resize + Normalize`，不引入随机增强。

## 5.4 训练实现

训练脚本 `demo-rps/dataprocess_modeltrain/train_a1_5class_classifier.py` 在启动时会先检查数据集目录，再校验 `train` 与 `val` split 中每个类别是否存在样本。核心校验逻辑如下：

```python
def verify_required_classes(split_name: str, samples_by_class):
    missing = [class_name for class_name in CLASS_NAMES if not samples_by_class.get(class_name)]
    if missing:
        raise RuntimeError(f"{split_name} split missing class data: {missing}")
```

该检查用于避免训练中途因类别缺失导致损失函数无法正常收敛或评估结果失真。

训练过程采用 AdamW 优化器和余弦退火学习率调度器：

```python
criterion = nn.CrossEntropyLoss(label_smoothing=args.label_smoothing)
optimizer = torch.optim.AdamW(model.parameters(), lr=args.lr, weight_decay=args.weight_decay)
scheduler = torch.optim.lr_scheduler.CosineAnnealingLR(optimizer, T_max=max(args.epochs, 1))
```

验证集指标优于历史最佳值时保存 `best.pt`，每轮训练都保存 `last.pt`。当验证集连续若干轮未达到最小增益阈值时，触发 early stopping：

```python
if epochs_without_improvement >= args.patience:
    print(f"Early stopping at epoch {epoch:03d} after {epochs_without_improvement} stale epochs")
    break
```

# 6. 模型评估与结果分析

## 6.1 核心评估指标

本项目评估指标覆盖整体准确率、训练稳定性和类别均衡性三个层面。训练脚本记录的核心指标包括：

- `train_loss`；
- `val_loss`；
- `train_top1`；
- `val_top1`；
- `test_top1`；
- 每类 `precision`；
- 每类 `recall`；
- 每类 `F1`；
- 混淆矩阵。

其中，`top1` 用于表示模型输出最大 logit 是否对应真实类别，混淆矩阵用于分析类别间误判关系，每类 F1 用于判断各类别识别是否均衡。

## 6.2 评估实现

训练脚本中通过 logits 最大值计算预测类别：

```python
def accuracy_from_logits(logits: torch.Tensor, targets: torch.Tensor):
    preds = logits.argmax(dim=1)
    return (preds == targets).float().mean().item()
```

混淆矩阵以真实类别为行、预测类别为列进行统计：

```python
def confusion_matrix(preds: torch.Tensor, targets: torch.Tensor, num_classes: int):
    matrix = torch.zeros((num_classes, num_classes), dtype=torch.int64)
    for target, pred in zip(targets.tolist(), preds.tolist()):
        matrix[target, pred] += 1
    return matrix
```

每类指标由混淆矩阵计算得到：

```python
precision = tp / (tp + fp) if tp + fp else 0.0
recall = tp / (tp + fn) if tp + fn else 0.0
f1 = 2 * precision * recall / (precision + recall) if precision + recall else 0.0
```

该评估方式能够同时反映整体性能和单类误判情况，适合用于中期报告和答辩展示。

## 6.3 结果文件生成

训练结束后，脚本统一生成 CSV 与图像报告。实现入口如下：

```python
def save_training_reports(output_dir: Path, history: list[dict], val_matrix: torch.Tensor, test_matrix: torch.Tensor, class_names: list[str]):

```

其中：

- `accuracy_loss_curve.png` 用于展示训练与验证趋势；
- `confusion_matrix_val.png` 用于展示验证集类别混淆情况；
- `confusion_matrix_test.png` 用于展示测试集类别混淆情况；
- `per_class_metrics.csv` 用于记录每类 precision、recall、F1 和 support。

## 6.4 分析重点

中期阶段模型评估不只关注最终准确率，还需要分析以下内容：

1. 训练集和验证集 loss 是否同步下降；
2. 验证集 top1 是否稳定；
3. 是否出现训练集提升但验证集停滞的过拟合现象；
4. `forward` 与 `stop` 是否存在明显混淆；
5. `NoTarget` 是否被误判为有效目标；
6. 每类 F1 是否均衡。

其中，`forward` 和 `stop` 的混淆会直接影响底盘动作，`NoTarget` 的误判会影响空场景安全性。因此，上述类别需要作为后续数据补充和训练优化重点。

## 7.1 算子支持约束

A1 端部署要求模型导出图中的算子落在当前支持范围内。本项目五分类模型采用轻量卷积结构，导出后主要使用以下算子：

## 7.2 避免使用的算子

导出阶段刻意避免下列不适合当前链路的算子：

- `Sub`；
- `Div`；
- `Softmax`；
- Transformer 结构相关算子。

导出脚本中明确使用固定输入形状和静态图，避免动态维度带来的转换风险：

```python
torch.onnx.export(
    export_model,
    dummy_input,
    str(output_path),
    input_names=["input"],
    output_names=["logits"],
    opset_version=args.opset,
    external_data=False,
    do_constant_folding=True,
    dynamic_axes=None,
)
```

导出后再次执行 Identity 节点清理，降低图中冗余节点数量，提升后续转换稳定性。

## 7.3 模型导出链路

导出脚本 `demo-rps/dataprocess_modeltrain/export_onnx.py` 采用“读取权重 → 恢复模型 → 构造静态输入 → 导出 ONNX → 清理图结构 → 保存结果”的固定流程。其关键约束如下：

PC 工具不经过 A1 中继，适合单独验证 STM32 自身是否能正确响应控制帧。

### 8.4.3 A1 工具

| 项目 | 说明 |
| --- | --- |
| 目录 | `tools/A1/` |
| 默认端口 | `6203` |
| 核心职责 | COM13 → A1_TEST → STM32 中继控制 |
| 主要能力 | A1_TEST 会话检查、手动命令发送、经 A1 中继做底盘联通测试 |
| 适用场景 | 验证 A1 板端到 STM32 的整链路控制 |

A1 工具用于验证“板端识别结果能否真正传到 STM32”，是板端算法与底盘联动之间的关键桥梁。

### 8.4.4 三套工具的分工关系

| 工具 | 验证对象 | 关注点 |
| --- | --- | --- |
| Aurora | A1 视觉与板端状态 | 预览、拍照、A1_TEST、分类快照 |
| PC | STM32 直连链路 | 串口控制、协议、遥测 |
| A1 | A1 端到 STM32 整链路 | 中继控制、板端语义转发 |

这三套工具不是重复功能，而是分别覆盖视觉侧、控制侧和整链路三个层次。

# 9. 构建与运行流程

## 9.1 模型训练

本项目的训练流程由 `demo-rps/dataprocess_modeltrain/` 中的脚本完成，输入为处理后的五分类图片数据集，输出为模型权重、元数据和评估报告。训练入口采用固定目录和固定类别顺序，确保结果可复现、可回溯、可用于后续导出。

训练阶段的核心动作包括：

- 校验数据集目录是否存在；
- 校验 train 与 val split 的类别完整性；
- 构建训练、验证与测试数据加载器；
- 使用 MobileNetV1 完成五分类训练；
- 保存 `best.pt` 与 `last.pt`；
- 生成 `results.csv`、混淆矩阵和每类指标。

训练脚本的输出配置已在代码中固定：

```python
best_checkpoint_path = output_dir / "best.pt"
last_checkpoint_path = output_dir / "last.pt"
```

## 9.2 模型部署

训练完成后，模型先导出为 ONNX，再转换为 A1 板端可用格式。导出脚本固定输出名、固定输入尺寸、固定类别顺序，并禁用动态维度：

```python
torch.onnx.export(
    export_model,
    dummy_input,
    str(output_path),
    input_names=["input"],
    output_names=["logits"],
    opset_version=args.opset,
    external_data=False,
    do_constant_folding=True,
    dynamic_axes=None,
)
```

导出阶段对输入尺寸有硬性约束：

```python
if image_size != MODEL_IMAGE_SIZE:
    raise RuntimeError(
        f"this export script only supports fixed image size {MODEL_IMAGE_SIZE}, got {image_size}"
    )
```

该限制保证导出结果与板端输入预处理完全一致，避免导出后图结构与板端实际输入不匹配。

## 9.3 固件构建

板端构建通过完整 EVB 构建脚本完成。脚本会先检查 SDK 根目录、build 脚本和发布脚本是否存在，再执行 SDK 重新打包和 `ssne_ai_demo` 重建。核心逻辑如下：

```bash
if [[ ${RUN_APP_ONLY} -eq 0 ]]; then
  echo "[build_complete_evb.sh] 先执行 SDK 发布构建，确保基础缓存存在"
  bash "${SDK_RELEASE}"
else
  if [[ ! -f "${SDK_ARTIFACT}" ]]; then
    fail "--app-only 需要已有基础 SDK 产物: ${SDK_ARTIFACT}"
  fi
fi

echo "[build_complete_evb.sh] 重建 ssne_ai_demo"
bash "${SDK_BUILD_APP}"

echo "[build_complete_evb.sh] 重新打包最终 EVB 镜像"
bash "${SDK_RELEASE}"
```

最终产物为 `zImage.smartsens-m1-evb`，并复制到 `output/evb/<timestamp>/` 与 `output/evb/latest/`。

构建脚本还会对 SDK 内的构建控制文件做 LF 规范化，避免 Windows 环境写入 CRLF 后引发构建脚本异常。这一处理与本仓库的 A1 SDK 集成环境直接相关。

## 9.4 上板验证

A1 板端运行脚本会先检查模型和 OSD 资源是否齐备，再启动主程序：

if [ ! -f ./app_assets/models/1cfd4504-c065-4698-9554-9e114f5bfd47_best.m1model ]; then

该检查说明板端运行不只依赖可执行文件，还依赖模型文件和 OSD 资源文件。上板验证需要同时确认以下内容：

# 11. 问题与优化方案

## 11.1 YOLO 方案效果不理想

### 问题

项目早期采用 YOLO 检测式方案进行探索，但上版测试中稳定性和部署匹配度未达到当前演示目标。其主要问题包括模型结构复杂、后处理链路较长、检测框还原增加联调复杂度，以及板端转换约束更严格。

### 优化

当前主线切换为五分类模型方案。该方案将目标直接归纳为固定语义类别，减少检测框和多阶段后处理步骤，提升了训练、导出、部署和联调的一致性。

## 11.2 小样本类别缺失

### 问题

早期按视频数量划分数据集时，小样本类别容易在某个 split 中缺失，尤其是 `NoTarget` 类。这会导致训练脚本在启动阶段因类别不完整而失败，或者让评估结果失去代表性。

### 优化

当前改为按抽样帧数量划分，并在训练前强制检查 train 与 val 的类别完整性：

```python
verify_required_classes("train", train_samples_by_class)
verify_required_classes("val", val_samples_by_class)
```

该方式能够提高类别覆盖率，减少训练前置失败概率。

## 11.3 过拟合风险

### 问题

在数据规模有限的情况下，模型容易出现训练集提升快、验证集提升有限的情况。该问题在小样本固定类别任务中较为常见。

### 优化

训练脚本采用以下策略降低过拟合风险：

- 数据增强；
- `Dropout`；
- `Label smoothing`；
- `Weight decay`；
- `Early stopping`。

相关参数已固定在训练脚本中：

```python
parser.add_argument("--weight_decay", type=float, default=1e-3)
parser.add_argument("--dropout", type=float, default=0.4)
parser.add_argument("--label_smoothing", type=float, default=0.1)
parser.add_argument("--patience", type=int, default=5)
parser.add_argument("--min_delta", type=float, default=0.002)
```

## 11.4 板端算子限制

### 问题

A1 端不适合承载复杂检测头、动态图结构和部分不兼容算子。若模型结构选择不当，导出后可能出现算子不支持或图结构不可转换问题。

### 优化

当前模型采用 MobileNetV1 轻量结构，并在导出阶段禁用动态维度，避免引入 `Sub`、`Div`、`Softmax` 等不适合当前链路的算子。导出脚本只保留静态输入、静态输出和固定类别顺序，降低转换风险。

## 11.5 调试路径混淆

### 问题

PC 直连 STM32 与 A1 中继 STM32 两条路径容易混用。如果没有明确区分，容易把底盘协议问题误判为板端问题，或者把板端问题误判为底盘协议问题。

### 优化

当前文档和工具已经明确两条路径的边界：

- PC 直连 STM32：用于验证底盘协议与串口通信；
- A1 中继链路：用于验证板端语义到 STM32 控制的整链路传递。

通过链路拆分，问题定位粒度更清晰，联调成本更低。

## 11.6 当前优化结论

本项目中期阶段的优化重点已经从“修复单点问题”转向“稳定工程链路”。当前优化方案均围绕数据稳定性、训练稳定性、导出稳定性和联调稳定性展开，目标是保证五分类主线在中期阶段具备可复现、可部署、可验证的工程能力。
