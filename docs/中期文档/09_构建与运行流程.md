# 9. 构建与运行流程

## 9.1 模型训练

本项目的训练流程由 `demo-rps/dataprocess_modeltrain/` 中的脚本完成，输入为处理后的五分类图片数据集，输出为模型权重、元数据和评估报告。训练入口采用固定目录和固定类别顺序，确保结果可复现、可回溯、可用于后续导出。

训练阶段的核心动作包括：

- 校验数据集目录是否存在；
- 校验 train 与 val split 的类别完整性；
- 构建训练、验证与测试数据加载器；
- 使用 MobileNetV1 完成五分类训练；
- 保存 `best.pt` 与 `last.pt`；
- 生成 `results.csv`、混淆矩阵和每类指标。

训练脚本的输出配置已在代码中固定：

```python
best_checkpoint_path = output_dir / "best.pt"
last_checkpoint_path = output_dir / "last.pt"
```

## 9.2 模型部署

训练完成后，模型先导出为 ONNX，再转换为 A1 板端可用格式。导出脚本固定输出名、固定输入尺寸、固定类别顺序，并禁用动态维度：

```python
torch.onnx.export(
    export_model,
    dummy_input,
    str(output_path),
    input_names=["input"],
    output_names=["logits"],
    opset_version=args.opset,
    external_data=False,
    do_constant_folding=True,
    dynamic_axes=None,
)
```

导出流程对输入尺寸有硬性约束：

```python
if image_size != MODEL_IMAGE_SIZE:
    raise RuntimeError(
        f"this export script only supports fixed image size {MODEL_IMAGE_SIZE}, got {image_size}"
    )
```

该限制保证导出结果与板端输入预处理完全一致，避免导出后图结构与板端实际输入不匹配。

## 9.3 固件构建

板端构建通过完整 EVB 构建脚本完成。脚本会先检查 SDK 根目录、build 脚本和发布脚本是否存在，再执行 SDK 重新打包和 `ssne_ai_demo` 重建。核心逻辑如下：

```bash
if [[ ${RUN_APP_ONLY} -eq 0 ]]; then
  echo "[build_complete_evb.sh] 先执行 SDK 发布构建，确保基础缓存存在"
  bash "${SDK_RELEASE}"
else
  if [[ ! -f "${SDK_ARTIFACT}" ]]; then
    fail "--app-only 需要已有基础 SDK 产物: ${SDK_ARTIFACT}"
  fi
fi

echo "[build_complete_evb.sh] 重建 ssne_ai_demo"
bash "${SDK_BUILD_APP}"

echo "[build_complete_evb.sh] 重新打包最终 EVB 镜像"
bash "${SDK_RELEASE}"
```

最终产物为 `zImage.smartsens-m1-evb`，并复制到 `output/evb/<timestamp>/` 与 `output/evb/latest/`。

构建脚本还会对 SDK 内的构建控制文件做 LF 规范化，避免 Windows 环境写入 CRLF 后引发构建脚本异常。这一处理与本仓库的 A1 SDK 集成环境直接相关。

## 9.4 上板验证

A1 板端运行脚本会先检查模型和 OSD 资源是否齐备，再启动主程序：

```sh
if [ ! -f ./app_assets/models/1cfd4504-c065-4698-9554-9e114f5bfd47_best.m1model ]; then
    echo "[APP] missing A1 5-class model"
    exit 1
fi

if [ ! -f ./app_assets/background.ssbmp ] || [ ! -f ./app_assets/background_colorLUT.sscl ]; then
    echo "[APP] missing OSD assets"
    exit 1
fi

./ssne_ai_demo
```

该检查说明板端运行不只依赖可执行文件，还依赖模型文件和 OSD 资源文件。上板验证需要同时确认以下内容：

- 摄像头输入是否正常；
- 分类结果是否正常；
- OSD 显示是否正常；
- UART 控制输出是否正常；
- 模型文件路径与脚本检查路径是否一致。
