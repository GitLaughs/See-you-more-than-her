# 7. A1 端部署兼容性

## 7.1 算子支持约束

A1 端部署要求模型导出图中的算子落在当前支持范围内。本项目五分类模型采用轻量卷积结构，导出后主要使用以下算子：

- `Conv`；
- `BatchNormalization`；
- `Relu`；
- `AveragePool`；
- `GlobalAveragePool`；
- `Flatten`。

该算子组合适合板端推理和后续模型转换，避免引入复杂检测头和额外后处理节点。

## 7.2 避免使用的算子

导出阶段刻意避免下列不适合当前链路的算子：

- `Sub`；
- `Div`；
- `Softmax`；
- Transformer 结构相关算子。

导出脚本中明确使用固定输入形状和静态图，避免动态维度带来的转换风险：

```python
torch.onnx.export(
    export_model,
    dummy_input,
    str(output_path),
    input_names=["input"],
    output_names=["logits"],
    opset_version=args.opset,
    external_data=False,
    do_constant_folding=True,
    dynamic_axes=None,
)
```

导出后再次执行 Identity 节点清理，降低图中冗余节点数量，提升后续转换稳定性。

## 7.3 模型导出链路

导出脚本 `demo-rps/dataprocess_modeltrain/export_onnx.py` 采用“读取权重 → 恢复模型 → 构造静态输入 → 导出 ONNX → 清理图结构 → 保存结果”的固定流程。其关键约束如下：

- 只支持 `320×320` 输入尺寸；
- 只支持固定五分类输出；
- 通过 `metadata.json` 读取训练参数；
- 导出输出名固定为 `input` 与 `logits`。

导出过程中的模型定义如下：

```python
self.backbone = timm.create_model(
    "mobilenetv1_100",
    pretrained=False,
    num_classes=0,
    global_pool="",
    in_chans=INPUT_CHANNELS,
)
```

该结构避免了分类头中引入不必要的算子分支，有利于板端转换。

## 7.4 部署流程

当前部署流程为：

```text
PyTorch 训练
  ↓
best.pt
  ↓
ONNX 导出
  ↓
best.onnx
  ↓
A1 模型转换
  ↓
.m1model
  ↓
放入 ssne_ai_demo/app_assets/models/
  ↓
完整 EVB 镜像构建
  ↓
上板验证
```

部署阶段需要同步检查三项内容：

1. 板端模型文件是否存在；
2. 板端运行脚本检查路径是否一致；
3. 类别顺序是否与训练、导出完全一致。

## 7.5 当前兼容性结论

当前五分类模型采用 MobileNetV1 作为骨干，输入为 320×320 单通道图像，输出为五类 logits，不依赖复杂检测后处理，也不依赖不兼容的动态算子。该模型具备进一步转换为板端可用模型的工程基础，并已满足中期阶段的部署验证前提。
