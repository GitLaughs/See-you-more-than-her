# 基于 SmartSens A1 视觉板的智能小车视觉交互系统中期技术报告

## 摘要

本项目面向嵌入式视觉感知与智能小车控制场景，设计并实现一套基于 SmartSens A1 视觉板和 STM32 控制板的视觉交互系统。系统以 A1 视觉板作为图像采集、AI 推理和状态显示核心，以 STM32 控制板作为运动控制核心，通过 UART 串口实现视觉识别结果到小车运动指令的传递。Windows 上位机工具用于摄像头预览、串口调试和联调验证。

中期阶段的主要目标是完成系统整体架构搭建，明确 A1 板端程序、Windows 调试工具、STM32 控制链路之间的分工，初步打通“摄像头输入—AI 识别—OSD 显示—UART 通信—小车动作”的演示闭环，并整理后续完善方向。

**关键词：** SmartSens A1；嵌入式视觉；AI 识别；OSD；UART；STM32；智能小车

## 1. 项目概述

### 1.1 项目背景

随着边缘 AI、嵌入式视觉和智能移动设备的发展，越来越多的智能终端开始具备本地图像感知、目标识别和自主控制能力。相比完全依赖云端计算的方案，边缘端视觉系统具有响应速度快、部署灵活、隐私性较好等优点，适合用于智能小车、服务机器人、巡检设备和教学实验平台等场景。

本项目选择 SmartSens A1 视觉板作为视觉感知核心，利用其摄像头输入、AI 推理、图像显示和外设通信能力，实现对周围环境或交互目标的感知；同时使用 STM32 控制板作为底层运动控制单元，负责执行小车前进、停止、转向等动作。通过 A1 与 STM32 之间的 UART 通信，将视觉识别结果转化为小车控制指令，从而形成一套完整的视觉交互智能小车系统。

### 1.2 项目目标

本项目的总体目标是设计并实现一套可演示、可调试、可扩展的视觉交互智能小车系统。主要目标包括：

1. 实现 A1 视觉板端摄像头图像采集与实时预览；
2. 在 A1 板端运行 AI 识别程序，完成对人、手势或指定状态的识别；
3. 使用 OSD 在预览画面中叠加识别结果、系统状态和动作提示；
4. 通过 UART 串口实现 A1 视觉板与 STM32 控制板之间的通信；
5. 由 STM32 根据 A1 下发的控制指令驱动小车完成基础动作；
6. 开发并整理 Windows 上位机调试工具，用于视频预览、串口调试和演示辅助；
7. 形成完整的构建、运行、测试和提交资料，为最终作品展示做准备。

### 1.3 中期阶段目标

根据中期作品提交要求，本阶段重点不是追求所有功能完全定型，而是完成系统主线搭建和关键功能验证。中期阶段目标包括：

1. 明确系统总体架构和软硬件分工；
2. 整理 A1 SDK、板端 AI demo、Windows 工具和构建脚本；
3. 验证 A1 板端程序的运行路径和图像预览流程；
4. 初步实现或验证 A1 到 STM32 的 UART 通信链路；
5. 设计视觉识别结果到小车动作的映射关系；
6. 规划 OSD 状态提示和演示动画；
7. 形成技术文档、PPT 和演示视频的素材基础。

## 2. 系统总体设计

### 2.1 总体架构

系统整体由视觉感知层、通信控制层、运动执行层和上位机调试层组成。

```text
摄像头模组
  ↓
SmartSens A1 视觉板
  ├─ 图像采集
  ├─ AI 推理识别
  ├─ OSD 状态显示
  └─ UART 控制指令发送
        ↓
      STM32 控制板
        ↓
  电机驱动 / 小车底盘 / 执行机构

Windows 上位机工具
  ├─ 摄像头预览
  ├─ COM13 串口终端
  ├─ A1_TEST 调试命令
  ├─ PC 直连 STM32 调试
  └─ A1 中转 STM32 控制验证
```

A1 视觉板负责完成图像输入、AI 推理、结果解析和可视化显示；STM32 控制板负责接收串口指令并控制小车运动；Windows 上位机主要用于开发调试阶段，帮助观察视频画面、发送测试命令、验证串口通信和记录演示素材。

### 2.2 硬件组成

| 硬件模块 | 主要作用 |
| --- | --- |
| SmartSens A1 视觉板 | 摄像头输入、AI 推理、OSD 显示、UART 指令发送 |
| 摄像头模组 | 采集实时图像，为 AI 识别提供输入 |
| STM32 控制板 | 接收 A1 指令，控制小车底盘和执行机构 |
| 小车底盘 | 执行前进、停止、转向等运动动作 |
| Windows 电脑 | 运行调试工具，进行视频预览和串口调试 |
| 串口连接 | 实现 PC、A1、STM32 之间的通信和调试 |

### 2.3 软件组成

| 软件模块 | 功能说明 |
| --- | --- |
| A1 板端 AI demo | 完成摄像头输入、模型推理、识别结果处理、OSD 显示和 UART 控制 |
| SDK / 固件构建脚本 | 编译 A1 应用并重新打包生成 EVB 固件镜像 |
| Aurora 工具 | 提供 Windows 端视频预览、COM13 串口终端和 A1 调试入口 |
| PC 调试工具 | 用于 PC 直接连接 STM32 进行串口控制验证 |
| A1 中转工具 | 通过 PC COM13 访问 A1，再由 A1 中转控制 STM32 |
| 项目文档 | 记录环境搭建、编译烧录、架构设计、测试和提交材料 |

### 2.4 数据流与控制流

系统运行时的数据流和控制流如下：

1. 摄像头采集实时图像并输入 A1 视觉板；
2. A1 板端 AI demo 对图像进行预处理和模型推理；
3. 推理结果经过解析后生成当前识别状态；
4. A1 根据识别状态更新 OSD 显示内容；
5. A1 根据识别状态生成小车控制指令；
6. 控制指令通过 UART 发送至 STM32；
7. STM32 解析指令后控制小车底盘完成动作；
8. Windows 上位机在开发阶段用于观察画面、发送命令和记录调试结果。

## 3. 关键技术路线

### 3.1 A1 板端视觉识别流程

A1 板端程序以 SDK 中的 AI demo 为基础，完成摄像头图像采集、AI 推理和识别结果处理。基本流程如下：

```text
摄像头采集图像
  ↓
图像预处理
  ↓
AI 模型推理
  ↓
输出类别、置信度和位置信息
  ↓
结果解析与状态判断
  ↓
触发 OSD 显示和小车控制逻辑
```

当前阶段主要关注板端 demo 的稳定运行和识别结果的可用性。后续将根据小车交互需求，将识别结果进一步映射到具体动作，例如识别到前进手势时发送前进指令，识别到停止手势时发送停止指令。

### 3.2 OSD 可视化显示设计

OSD 显示用于在视频画面上直接反馈当前系统状态，使演示效果更加直观。OSD 内容计划包括：

| 系统状态 | OSD 显示内容 |
| --- | --- |
| 识别到人 | 显示欢迎提示或人物检测提示 |
| 前进手势 | 显示小车前进图标或动画 |
| 停止手势 | 显示停止提示 |
| 避障状态 | 显示避障告警或绕行动画 |
| 空闲状态 | 显示系统待机或识别中提示 |

OSD 的作用不仅是美化画面，也可以辅助判断板端 AI 程序是否正在运行、识别状态是否发生切换以及控制逻辑是否被触发。

### 3.3 A1 与 STM32 串口通信设计

A1 与 STM32 之间采用 UART 串口通信。A1 根据 AI 识别结果生成控制指令，STM32 接收后执行对应动作。中期阶段优先验证通信链路的连通性和基础指令的可靠性。

初步指令设计如下：

| 指令 | 含义 | STM32 执行动作 |
| --- | --- | --- |
| `FORWARD` | 前进 | 小车向前运动 |
| `STOP` | 停止 | 小车停止运动 |
| `LEFT` | 左转 | 小车向左转向 |
| `RIGHT` | 右转 | 小车向右转向 |
| `AVOID` | 避障 | 执行避障或绕行动作 |

后续可以根据实际通信稳定性，将简单字符串协议扩展为带帧头、命令字、校验位和结束符的结构化协议，以提高抗干扰能力和解析可靠性。

### 3.4 STM32 小车控制设计

STM32 控制板作为运动执行核心，主要负责以下任务：

1. 初始化串口、电机驱动和相关外设；
2. 接收来自 A1 的 UART 控制指令；
3. 解析指令并映射为底盘运动动作；
4. 控制小车完成前进、停止、左转、右转等基础动作；
5. 后续接入避障传感器或状态反馈模块，提高自主运行能力。

在系统分工上，A1 负责“看见和判断”，STM32 负责“执行和控制”。这种分层方式可以减少单个模块的复杂度，也便于分别调试视觉算法和底盘控制逻辑。

### 3.5 Windows 上位机调试设计

Windows 上位机工具主要用于开发和演示阶段，包含三类能力：

1. 视频预览：用于观察摄像头画面和 OSD 显示效果；
2. 串口终端：用于访问 A1 的调试命令入口；
3. 控制验证：用于区分 PC 直连 STM32 和 PC 通过 A1 中转控制 STM32 两种路径。

在联调过程中，需要明确两条控制链路：

```text
PC 直连 STM32：PC 串口 → STM32 UART
A1 中转控制：PC COM13 → A1 A1_TEST → A1 UART → STM32 UART
```

明确这两条路径可以避免调试时将上位机命令、A1 中转命令和 STM32 直接命令混淆。

## 4. 当前实现进展

### 4.1 A1 板端程序进展

当前项目已经整理出 A1 板端运行主线，板端 AI demo 位于 SmartSens SDK 的应用 demo 目录中。该 demo 是当前系统视觉识别、OSD 显示和 A1 侧调试逻辑的主要承载位置。

中期阶段已完成或正在推进的工作包括：

2. 整理完整 EVB 镜像构建流程；
3. 确认单独编译 demo 与完整固件打包之间的关系；

5. 为后续演示准备识别状态、动作提示和控制命令的映射关系。

### 4.2 构建与固件打包进展

项目使用 Docker 构建环境完成 A1 SDK 和 EVB 固件构建。完整固件构建流程包括 SDK 基础层构建、应用 demo 编译和最终镜像打包。由于板端 app 需要进入最终 EVB 镜像，因此仅编译应用程序本身不能作为最终可烧录产物，还需要重新执行固件打包流程。

当前已经整理的构建方向包括：

1. 使用脚本统一触发完整 EVB 构建；
2. 支持在已有缓存基础上进行 app-only 构建；
3. 输出最终可烧录镜像；
4. 将构建、烧录和运行检查流程写入项目文档。

### 4.3 Windows 调试工具进展

项目已整理 Windows 端辅助工具，用于视频预览和串口调试。工具主要分为三类：

1. Aurora 工具：用于摄像头预览、COM13 串口终端和 A1 调试；
2. PC 工具：用于 PC 直接连接 STM32 进行底层控制测试；
3. A1 工具：用于 PC 经由 A1 中转向 STM32 发送控制命令。

当前阶段已明确不同工具的职责边界，避免将 PC 直连 STM32 与 A1 中转 STM32 两种控制路径混用。

### 4.4 A1 与 STM32 通信进展

A1 与 STM32 的 UART 通信是当前项目联调的关键。系统目标是由 A1 根据视觉识别结果向 STM32 发送控制指令，STM32 再驱动小车完成动作。

当前阶段重点围绕以下内容推进：

1. 确认 A1 到 STM32 的 UART 连接方式；
2. 验证 STM32 UART3 相关引脚通信；
3. 设计基础控制指令格式；
4. 通过 PC 直连和 A1 中转两种方式分别验证 STM32 响应；
5. 为后续视觉识别触发小车动作做准备。

## 6. 当前问题与优化方向

### 6.1 当前存在问题

中期阶段系统仍处于集成和联调过程中，主要问题包括：

1. A1 与 STM32 的 UART 通信链路仍需要进一步稳定验证；
2. 视觉识别结果可能存在抖动，需要设计状态保持或防抖机制；
3. OSD 显示资源和动画效果仍需进一步美化；
4. 小车控制动作目前以基础动作验证为主，复杂运动策略尚未完成；
5. 固件构建链路较长，对环境和缓存状态有一定依赖；
6. 上位机调试工具功能较多，最终演示流程还需要进一步简化；
7. 测试数据、截图和视频素材还需要在后续联调过程中持续补充。

### 6.2 后续优化方向

后续阶段计划从功能完善、稳定性提升和展示效果三个方面继续推进：

1. 完善 AI 识别类别与小车动作之间的映射关系；
2. 增加识别结果防抖机制，减少误触发；
3. 优化 OSD 图标、状态气泡和动作动画，使演示效果更加直观；
