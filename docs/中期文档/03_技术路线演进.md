# 3. 技术路线演进

## 3.1 早期 YOLO 检测方案

项目早期曾尝试基于 YOLO 的目标检测方案，希望通过检测框识别目标位置和类别，再将检测结果映射到 OSD 显示与底盘控制。该方案的优势是表达能力强，能够同时输出目标类别和位置，适合通用目标检测类任务。

但在当前项目场景中，YOLO 方案也带来明显问题：

- 模型结构更复杂，板端转换和算子约束更严格；
- 输入输出后处理链路较长；
- 检测框还原、阈值筛选和类别映射增加联调复杂度；
- 上版测试效果不理想，演示稳定性不足。

因此，YOLO 方案更适合作为前期探索和技术储备，不适合作为中期展示主线。

## 3.2 YOLO 方案在报告中的定位

YOLO 方案应写入报告，但只作为项目早期探索和路线迭代记录，不作为当前代码展示重点。报告中建议只说明方案尝试、测试反馈和切换原因，不展开具体检测网络的代码细节，以免冲淡当前五分类主线。

推荐表述如下：

> 项目前期曾尝试基于 YOLO 的检测式方案，但上版测试中稳定性和部署匹配度未达到当前演示目标，因此中期阶段将主线调整为轻量五分类模型方案。YOLO 方案作为前期探索和方案演进记录保留，不作为当前展示代码重点。

## 3.3 当前五分类模型主线

当前主线采用轻量五分类模型，将视觉状态直接划分为 `person`、`stop`、`forward`、`obstacle` 和 `NoTarget`。相比 YOLO 检测方案，五分类方案更适合当前中期演示，主要体现在以下几方面：

- 输出语义直接，便于映射动作；
- 推理链路短，部署更轻量；
- 不需要检测框坐标还原；
- 更容易满足 A1 算子支持范围；
- 训练、评估、导出和部署流程更稳定。

这条技术路线的核心不是追求更复杂的视觉能力，而是围绕“能稳定识别、能稳定控制、能稳定展示”三个目标收敛。

## 3.4 版本演进脉络

项目提交历史跨度较长，当前可见提交已覆盖 300+ 次变更。整体演进不是单线推进，而是多个模块并行收敛，最后在 A1 五分类主线上完成闭环。按阶段看，演进可以分成几条清晰脉络。

### 3.4.1 仓库初始化与基础设施阶段

最早阶段主要完成仓库落地、SDK 导入、CI 和 gitignore 规范、基础构建入口整理。这一阶段的目标不是做出完整演示，而是先把工程骨架搭起来，保证后续能持续集成、能重复构建、能追踪变更。

这一阶段的特点是：

- SDK 树逐步导入并整理；
- 构建脚本和 Docker 环境开始成形；
- `.gitignore`、文档骨架和 CI 流程开始稳定；
- `ssne_ai_demo`、`demo_app` 等目录命名和位置逐步明确。

### 3.4.2 早期功能探索阶段

随后项目开始围绕图像显示、驱动适配、ROS/底盘集成和基础联通测试做探索。这个阶段的提交历史里可以看到：

- 图像 dump 和显示相关能力逐步增加；
- UART、GPIO、底盘驱动和 `libssne` 版本不断适配；
- 文档从零散记录逐步变成可阅读的项目说明；
- 早期 ROS/外设相关路径仍然保留，作为集成试验区存在。

这一阶段的本质是“广撒网”：先把 A1 能接什么、能跑什么、哪些模块会冲突摸清楚，再决定主线。

### 3.4.3 构建链路与文档体系成型阶段

到 3 月中下旬，提交历史明显开始向“能构建、能复现、能交付”集中。这个阶段的关键变化包括：

- 完整 EVB 构建链路被统一；
- Docker / build 脚本逐步稳定；
- README、快速上手、环境搭建、构建烧录等文档开始成体系；
- 文档从单页记录升级成完整中期材料和提交说明。

这一阶段说明项目已经从“能跑的零散代码”转向“能交的工程项目”。

### 3.4.4 Aurora 与 Windows 联调工具扩展阶段

4 月开始，Windows 工具成为项目推进主线之一。Aurora 从单纯视频预览逐步扩展为：

- 摄像头预览；
- 拍照和训练集采样；
- COM13 串口终端；
- A1_TEST 手动调试；
- A1 视觉预览和状态查看；
- 与板端、底盘联动的测试界面。

这一阶段的提交历史非常密集，说明项目不再只是改板端代码，而是在同时打通“视觉输入—调试界面—串口通信—联调验证”的整条链路。

### 3.4.5 视觉主线切换阶段

4 月中旬到 5 月初，视觉方案经历了几次明显迭代：

- 从早期检测式方案探索出发；
- 到 YOLOv8 head6 检测方案；
- 再到当前五分类分类方案收敛。

这条主线背后反映的是：

- 目标从“做复杂检测”转为“做稳定识别与控制”；
- 部署约束从“通用检测网络”转为“A1 可落地算子”；
- 结果输出从“检测框 + 后处理”转为“直接类别 + 动作映射”。

也就是说，提交历史不是简单的模型替换，而是项目目标逐步从探索型方案收敛到可展示、可部署、可验收方案的过程。

### 3.4.6 训练链路完善阶段

5 月初训练脚本逐步补全，五分类数据处理、训练、验证、测试和导出形成闭环。这个阶段最重要的变化包括：

- 训练输入从视频探索转为稳定图片数据集；
- 数据集划分从按视频改为按帧；
- 训练脚本增加正则化、早停和完整报告输出；
- 导出流程开始围绕 A1 算子兼容性约束来设计。

这一阶段把“有模型”变成“有可复现训练流程的模型”。

### 3.4.7 当前收敛阶段

最新提交已经把主线收敛到：

- 五分类模型；
- 训练输出和评估报告；
- A1 板端资源和运行脚本；
- Aurora / PC / A1 三套测试工具；
- 中期文档和提交材料。

当前项目的演进重点不再是继续堆新功能，而是把现有链路稳定下来，把证据、报告和交付材料补齐。

### 3.4.8 各模块演进总结

| 模块 | 早期状态 | 演进过程 | 当前状态 |
| --- | --- | --- | --- |
| SDK / 构建 | SDK 初始导入，构建方式分散 | Docker、build 脚本、EVB 打包逐步统一 | 以完整 EVB 构建和 `ssne_ai_demo` 打包为主 |
| 板端 Demo | 以厂商 demo 和基础图像链路为起点 | 经历人脸检测、YOLO、OSD、UART、分类器多轮迭代 | 当前主线为五分类推理 + OSD + UART 控制 |
| AI 模型 | YOLO 和检测式方案探索 | 从检测框方案转向轻量分类方案 | MobileNetV1 五分类模型成为当前主线 |
| 数据处理 | 视频和图片处理流程不稳定 | 增加抽帧、裁剪、划分和训练脚本 | `processed_dataset_regularized` 支撑稳定训练 |
| Windows 工具 | 单一预览和串口能力 | Aurora 扩展后又拆分 PC/A1 工具 | 三套工具分别覆盖视觉、直连、整链路 |
| STM32 联调 | 串口和底盘协议探索 | PC 直连与 A1 中继链路逐步分开 | 两条链路边界清晰，便于定位问题 |
| 文档体系 | 零散说明和单页记录 | 快速上手、构建、架构、中期材料不断补齐 | 已形成中期报告大纲和专项训练文档 |

## 3.5 小结

技术路线的收敛结果很明确：项目已经从复杂检测探索转向轻量分类主线，从单点功能验证转向完整工程闭环。当前的主线不再是证明“能不能做视觉识别”，而是证明“能否稳定地在 A1 端完成识别、显示、控制和联调”。
