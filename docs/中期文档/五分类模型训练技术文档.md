# 五分类模型训练技术文档

## 1. 项目背景

本项目面向 SmartSens A1 视觉板端部署场景，构建了一套五分类轻量级图像分类模型，用于识别视觉交互中的核心状态。模型训练完成后，既可在板端完成实时推理，也可作为后续 A1 端行为映射、OSD 状态显示和底盘控制逻辑的基础。

与传统目标检测方案相比，五分类方案更适合当前评委演示主线：

- 输入和推理链路更短，部署更轻量；
- 输出语义更清晰，便于直接映射控制逻辑；
- 模型结构更容易满足 A1 侧算子支持范围；
- 在样本数量有限的情况下，比复杂检测网络更容易稳定训练。

当前五分类类别定义为：

1. `person`
2. `stop`
3. `forward`
4. `obstacle`
5. `NoTarget`

其中，`person`、`stop`、`forward`、`obstacle` 用于表达项目中的典型视觉状态，`NoTarget` 作为空场景或无有效目标状态，参与统一训练与推理。

---

## 2. 设计目标

五分类模型训练的目标不是追求通用大模型效果，而是为 A1 板端演示链路提供一个稳定、轻量、可部署、可调试的分类器。具体目标如下：

### 2.1 功能目标

- 实现五类图像状态识别；
- 支持训练、验证、测试三阶段数据划分；
- 支持训练过程自动保存模型权重；
- 支持导出 ONNX 模型；
- 支持记录训练曲线、混淆矩阵和每类精度指标；
- 支持后续转换为 A1 可部署格式。

### 2.2 工程目标

- 训练脚本可重复执行，结果可追溯；
- 数据预处理流程固定，避免训练与部署不一致；
- 训练输出包括权重文件、元数据和可视化报告；
- 导出模型满足 A1 端支持的算子范围；
- 训练流程便于后续扩展到更多数据或更长训练轮次。

## 3. 数据集来源与组织方式

### 3.1 数据来源

训练数据来源于项目采集的五类视频素材，统一存放在 `data/rps_dataset/` 目录下。原始数据按类别组织，每类包含若干视频文件，视频内容覆盖不同角度、不同光照和不同拍摄距离下的目标状态。

### 3.2 数据处理流程

训练前并不直接使用视频，而是先将视频转换为帧图片，再构建训练集。完整流程如下：

1. 从视频逐帧读取图像；
2. 按固定帧间隔抽帧，降低样本冗余；
3. 将抽取帧统一保存为图片；
4. 按类别和帧数划分为 `train / val / test`；
5. 训练脚本直接读取处理后的图片目录。

当前使用的数据处理结果目录为：

```text
data/rps_dataset/processed_dataset_regularized/
```

目录结构如下：

```text
data/rps_dataset/processed_dataset_regularized/
  train/
    person/
    stop/
    forward/
    obstacle/
    NoTarget/
  val/
    person/
    stop/
    forward/
    obstacle/
    NoTarget/
  test/
    person/
    stop/
    forward/
    obstacle/
    NoTarget/
```

### 3.3 数据划分原则

数据划分采用按帧数划分的方式，而不是单纯按视频数量划分。这样做的原因是：

- 某些类别的视频数量较少；
- 如果按视频数切分，可能出现某些 split 中缺少某一类别；
- `NoTarget` 类样本最容易受此影响；
- 按帧划分后，train/val/test 更容易同时覆盖全部类别。

这一改动可以显著降低训练前就因数据划分不均导致的失败概率。

## 4. 模型结构设计

### 4.1 模型总体结构

当前五分类模型采用轻量级 MobileNetV1 作为主干，输入单通道灰度图，输出 5 类分类 logits。模型整体由两部分组成：

- Backbone：`mobilenetv1_100`
- Head：1×1 卷积分类头 + 池化 + Flatten

对应训练脚本位于：

```text
demo-rps/dataprocess_modeltrain/train_a1_5class_classifier.py
```

### 4.2 输入形式

- 输入通道：`1`
- 输入尺寸：`320×320`
- 输入类型：灰度图

使用灰度图的原因：

- 摄像头场景中颜色信息并非主要判别依据；
- 灰度输入更省显存和计算；
- 更利于板端部署；
- 更容易控制训练和推理的一致性。

### 4.3 输出形式

模型输出 5 维 logits，对应五个类别：

```text
person / stop / forward / obstacle / NoTarget
```

训练时采用交叉熵损失进行单标签多分类训练。

### 4.4 训练脚本主要参数

默认训练参数如下：

| 参数 | 默认值 | 说明 |
| --- | --- | --- |
| `dataset_dir` | `data/rps_dataset/processed_dataset` | 数据集根目录 |
| `output_dir` | `outputs/a1_5class_mobilenetv1` | 输出目录 |
| `batch_size` | `64` | 批大小 |
| `epochs` | `30` | 训练轮数 |
| `lr` | `3e-4` | 学习率 |
| `weight_decay` | `1e-3` | 权重衰减 |
| `head_hidden_dim` | `128` | 分类头隐藏通道数 |
| `dropout` | `0.4` | Dropout 比例 |
| `label_smoothing` | `0.1` | 标签平滑 |
| `patience` | `5` | 早停耐心轮数 |
| `min_delta` | `0.002` | 最小提升阈值 |

## 5. 训练方法与实现思路

### 5.1 训练流程

训练脚本的主流程如下：

1. 读取训练、验证、测试集目录；
2. 校验 train 和 val 中每个类别是否存在样本；
3. 构建训练增强和验证增强；
4. 初始化 MobileNetV1 分类模型；
5. 使用 AdamW 作为优化器；
6. 使用余弦退火学习率调度；
7. 逐轮训练并验证；
8. 保存 best/last 权重；
9. 在训练结束后输出测试集结果和训练报告；
10. 按需导出 ONNX。

### 5.2 数据增强策略

训练阶段使用较强的数据增强，以提高模型对拍摄位置、光照变化和局部遮挡的鲁棒性。增强策略包括：

- Resize + RandomResizedCrop
- RandomColorJitter
- RandomRotation
- RandomHorizontalFlip
- RandomErasing
- Normalize

验证和测试阶段只做确定性的 Resize + Normalize，不使用随机增强。

### 5.3 正则化策略

为缓解过拟合，训练脚本中引入了多种正则化手段：

- `weight_decay=1e-3`
- `dropout=0.4`
- `label_smoothing=0.1`
- 数据增强
- 早停机制 `patience=5`

这些策略适合样本量不算特别大、类别数固定、部署目标明确的轻量分类任务。

### 5.4 训练监控与保存

训练过程中会保存以下内容：

- `best.pt`：验证集指标最优的模型权重；
- `last.pt`：最后一个 epoch 的模型权重；
- `metadata.json`：训练超参数与数据规模信息；
- `results.csv`：每轮训练的 loss 与 top1；
- `accuracy_loss_curve.png`：训练/验证曲线图；
- `confusion_matrix_val.png`：验证集混淆矩阵；
- `confusion_matrix_test.png`：测试集混淆矩阵；
- `per_class_metrics.csv`：每类 precision / recall / F1 / support。

这些输出可以直接用于中期答辩中的性能说明，也方便后续持续训练时比较不同实验版本。

## 6. 训练结果与性能评估

### 6.1 训练指标

训练脚本记录的核心指标包括：

- `train_loss`
- `val_loss`
- `train_top1`
- `val_top1`
- `test_top1`

同时还会输出混淆矩阵和每类指标，便于分析哪些类别容易混淆。

### 6.2 评估方式

本项目不只看最终准确率，还会从以下角度评估模型：

1. **整体准确率**：验证集 / 测试集 top1；
2. **收敛趋势**：训练损失和验证损失是否同步下降；
3. **稳定性**：验证精度是否出现大幅波动；
4. **类别均衡性**：每一类 precision、recall、F1 是否均衡；
5. **混淆情况**：例如 `forward` 是否被误判为 `stop`，`NoTarget` 是否与其他类混淆。

### 6.3 当前输出形式

当前脚本已经支持生成可视化训练结果，形式上接近常见训练框架的报告产物，但更适合本项目的分类任务：

- 曲线图代替单纯日志；
- 混淆矩阵代替单一准确率；
- 每类 F1 指标用于分析类别间差异。

### 6.4 中期展示时建议写法

如果需要放入中期报告正文，可以用如下方式描述当前效果：

- 模型已完成从数据处理到训练、验证、测试、导出的完整闭环；
- 训练过程能够稳定输出 best/last 权重和完整训练日志；
- 训练结果可通过曲线图、混淆矩阵和每类指标进行分析；
- 已具备部署到 A1 端进一步转换和验证的基础。

如果需要填写具体数值，可从 `results.csv` 和 `per_class_metrics.csv` 中直接摘取。

## 7. A1 端部署兼容性

### 7.1 算子兼容性要求

为保证模型能够部署到 A1，导出后的 ONNX 图必须满足 A1 支持的算子范围。当前项目确认过的安全算子主要包括：

- `Conv`
- `AveragePool` / `GlobalAveragePool`
- `BatchNormalization`
- `Relu`
- `Add`
- `Mul`
- `Concat`
- `Split`
- `Transpose`
- `Resize`（nearest）
- `ConvTranspose2d`
- `Upsample`

不建议或不支持的算子包括：

- `Sub`
- `Div`
- `Softmax`
- Transformer 架构相关算子

### 7.2 当前模型的兼容性结论

当前五分类模型采用 MobileNetV1 + 轻量分类头，导出后不会引入 Transformer、Softmax 或除法等不适合 A1 侧的复杂算子。训练脚本和导出测试也已经验证，导出图的主要节点都落在 A1 的安全算子范围内。

这意味着该模型具备进一步转换成板端可用模型的工程基础。

### 7.3 部署链路

训练完成后，部署链路为：

1. 训练得到 `best.pt`；
2. 导出 `best.onnx`；
3. 进行后续格式转换；
4. 生成板端模型文件；
5. 放入 A1 demo 的模型目录；
6. 在板端验证输入预处理、类别顺序和输出映射是否一致。

## 8. 已解决问题与优化方案

### 8.1 数据划分不均导致类别缺失

**问题：**
早期若按视频数量划分数据，容易出现某些 split 中缺少某个类别，尤其是 `NoTarget` 样本较少时，训练会直接失败。

**优化：**
改为按帧数划分数据，保证 train/val/test 中尽量同时覆盖所有类别。

### 8.2 训练过程过拟合

**问题：**
模型容量相对数据规模偏大时，训练集准确率上升快，但验证集提升有限，容易过拟合。

**优化：**
- 增强训练数据增强；
- 引入 `weight_decay`；
- 引入 `dropout`；
- 使用 `label_smoothing`；
- 加入早停。

### 8.3 训练结果不可视化

**问题：**
只看日志不利于分析训练过程，尤其不方便答辩展示。

**优化：**
新增训练报告输出：

- `results.csv`
- 曲线图
- 混淆矩阵
- 每类 precision / recall / F1

### 8.4 部署算子不确定

**问题：**
如果训练模型结构过复杂，ONNX 导出后可能包含 A1 不支持的算子。

**优化：**
使用轻量级 MobileNetV1，并在导出测试中检查算子集合，避免引入 `Softmax`、`Sub`、`Div`、Transformer 等不兼容结构。
