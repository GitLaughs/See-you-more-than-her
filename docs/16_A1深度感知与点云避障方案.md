# A1 深度感知与点云避障方案

> 目标: 围绕当前 A1 开发板摄像头链路，给出可落地的 3D 深度 / 点云 / 避障方案。不要求立即在深度模块写代码，重点是路线选择与集成方式。

---

## 一、当前基础与约束

当前项目已经具备以下条件:

- A1 板端摄像头采集: 1280 × 720
- 板端 YOLOv8 推理输入: 640 × 360
- 现有目标类别: person / forward / stop / obstacle_box
- 现有 ROS 可直接复用链路:
  - turn_on_wheeltec_robot 的 wheeltec_robot_node
  - wheeltec_multi 的 multi_avoidance

当前约束也很明确:

- 现有 SC132GS 链路本质上是单目 2D 图像，不自带真实深度
- 单目 2D 检测框只能做“相对远近”估计，不能天然得到高可信米制距离
- A1 板端算力更适合轻量推理，不适合直接堆大型深度网络 + 稠密点云全流程
- 真正用于“紧急避障”的方案，必须优先保证延迟和稳定性，而不是追求最漂亮的点云

因此，深度方案不建议一步到位直接做“板端稠密点云 + 实时建图”，而应该分阶段推进。

---

## 二、可行方案对比

### 方案 1: 2D 检测框 + 伪距离 + ROS 避障输入

核心思路:

- 继续使用当前 YOLO 检测 obstacle_box
- 用框宽、框高、框中心偏移估计“障碍物大致方向 + 近远等级”
- 将结果映射为 ROS 话题 object_tracker/current_position
- 复用现有 multi_avoidance 节点做避障修正

优点:

- 开发成本最低
- 不需要新增硬件
- 可以立刻与现有 wheeltec_multi 对接
- 适合先把“检测到障碍物 -> 机器人不再硬撞”跑通

缺点:

- 只有启发式距离，不是真深度
- 不适合窄缝穿越、台阶判断、复杂三维空间避障
- 点云只能做伪点云展示，不适合高可信导航

适用场景:

- 当前阶段最适合先落地
- 作为 ROS 系统调试的第一版视觉避障入口

结论:

- 推荐作为短期方案
- 重点是先把“视觉检测 -> ROS 避障动作链”稳定下来

---

### 方案 2: 单目深度估计 + 伪点云

核心思路:

- 使用单目深度网络从单帧图像回归深度图
- 常见模型可以是 MiDaS、Depth Anything 一类轻量版本
- 将深度图投影为伪点云，再供局部避障或可视化使用

优点:

- 不需要改硬件
- 可以生成较完整的深度图和伪点云
- 比单纯检测框更接近真实 3D 信息

缺点:

- 深度通常是相对深度，不一定是绝对米制深度
- 需要额外标定或尺度恢复，才能用于可靠停车阈值
- 板端实时运行压力较大，更适合先在 PC/ROS 侧验证

适用场景:

- 适合做“视觉深度感知”的第二阶段
- 可以先在 PC 侧 ROS 节点运行，再决定是否迁移到 A1

结论:

- 推荐作为中期方案
- 先做 PC 侧验证，再决定是否裁剪到板端 ROI 深度估计

---

### 方案 3: 双目相机 / 双摄标定 + 立体匹配

核心思路:

- 增加第二路摄像头
- 完成立体标定、极线校正、视差估计
- 通过视差直接恢复米制深度
- 输出深度图与真实点云

优点:

- 深度更真实，适合真正的局部避障和点云建图
- 比单目相对深度更容易得到可解释的米制距离
- 适合后续接入 ROS 点云链路

缺点:

- 需要新增硬件与同步/标定工作
- 算法链更长，调试量明显增加
- 对安装精度、基线长度、镜头畸变校正要求更高

适用场景:

- 真正想把“摄像头三维点云 + 避障”做扎实时，这是更正规路线

结论:

- 如果项目后面要做持续导航或复杂避障，双目比单目更值得投入

---

### 方案 4: ToF / 深度相机外设融合

核心思路:

- 外挂 ToF 或深度相机
- 摄像头继续负责识别 forward / stop / obstacle_box
- 深度传感器负责提供真实距离或点云
- 在 ROS 层做融合决策

优点:

- 深度可靠，避障鲁棒性高
- 算法开发难度通常低于从单目硬抠真实深度
- 非常适合快速验证导航与避障

缺点:

- 成本更高
- 硬件体积、供电、驱动兼容性要单独处理
- 对“只用 A1 自带摄像头”这个目标不完全纯粹

适用场景:

- 如果目标是尽快把避障可靠性做出来，这是工程上最稳的路线之一

结论:

- 如果预算允许，属于高性价比工程解法

---

### 方案 5: VSLAM / SfM 稀疏点云

核心思路:

- 依赖相机运动恢复空间结构
- 生成稀疏或半稠密地图
- 适合作为定位与环境建模的一部分

优点:

- 可以把视觉扩展到建图和长期定位
- 能得到更完整的空间结构表示

缺点:

- 不适合做短时紧急避障主链
- 依赖运动、纹理和光照
- 系统复杂度很高

适用场景:

- 更适合长期路线中的建图/定位能力，而不是当前阶段的障碍物紧急处理

结论:

- 不建议作为当前阶段主线方案

---

## 三、推荐路线

### 阶段 1: 先把 2D 视觉避障链跑通

建议目标:

- PC 摄像头检测到 obstacle_box
- 将障碍物方向和伪距离映射到 object_tracker/current_position
- 使用 multi_avoidance 修正 cmd_vel_ori
- wheeltec_robot_node 输出最终 cmd_vel 到 STM32

这一阶段的价值:

- 不需要深度硬件
- 已经能完成“视觉触发 ROS 避障”
- 为后续深度感知留下固定接口

这一阶段本质上是:

- 用 obstacle_box 先替代真正深度传感器
- 把 ROS 避障链路打穿

---

### 阶段 2: 在 PC 侧接入单目深度或外部深度设备

建议目标:

- 保持 A1 板端仍做轻量检测与基础控制
- 在 PC / ROS 侧增加深度图生成
- 将深度 ROI 或伪点云转化为局部障碍物信息

推荐做法:

- 单目深度: 先在 PC 上验证
- ToF / 深度相机: 若追求稳定避障，可优先选这条

此阶段输出可统一为:

- /camera/depth/image
- /camera/points
- /object_tracker/current_position

---

### 阶段 3: 再考虑 A1 板端轻量深度感知

建议目标:

- 不是整帧深度图
- 而是只对 obstacle_box ROI 做轻量深度估计或距离等级判断
- 输出近/中/远三档，再与板端控制逻辑耦合

这样做的原因:

- 板端最需要的是低延迟避障决策，不是漂亮的全图点云
- ROI 深度比整图深度更节省算力

---

## 四、建议的 ROS 话题与模块分工

### 4.1 当前阶段推荐

- 摄像头检测:
  - 输出 forward / stop / obstacle_box
- 运动控制:
  - /cmd_vel 或 /cmd_vel_ori
- 视觉避障输入:
  - /object_tracker/current_position

建议分工:

- YOLO 检测模块:
  - 负责识别类别和检测框
- 视觉到 ROS 映射层:
  - 将 obstacle_box 框映射为 angle_x + distance
- multi_avoidance:
  - 负责把 cmd_vel_ori 修正为 cmd_vel
- wheeltec_robot_node:
  - 负责把 cmd_vel 发到 STM32

### 4.2 深度阶段推荐

- 深度图:
  - /camera/depth/image
- 点云:
  - /camera/points
- 局部障碍物:
  - /object_tracker/current_position 或自定义 obstacle_array

---

## 五、点云实现建议

如果只是为了“有 3D 点云可视化”，建议优先做以下两种:

### 方式 A: 单目深度图投影为伪点云

- 输入: RGB/灰度图 + 单目深度图
- 输出: PointCloud2
- 适合调试展示和相对深度分析

特点:

- 视觉效果好
- 工程难度适中
- 但绝对距离可信度一般

### 方式 B: 双目/ToF 真实深度点云

- 输入: 真实深度或视差
- 输出: PointCloud2
- 适合后续真正接入局部避障/建图

特点:

- 更适合长期演进
- 但需要额外硬件或更复杂标定

---

## 六、明确建议

如果只看当前项目阶段，最合理的建议是:

1. 短期先用 obstacle_box + 伪距离触发现有 ROS 避障链，不做真正深度模块开发。
2. 中期把单目深度放到 PC/ROS 侧验证，先做深度图和伪点云，不急着上板端。
3. 如果后续目标是可靠避障和点云导航，优先考虑双目或 ToF，而不是继续强压单目绝对深度。
4. A1 板端后续更适合做轻量 ROI 深度/距离等级判断，而不是整图稠密点云主流程。

---

## 七、落地优先级

### 优先级 A

- 现有 YOLO obstacle_box -> object_tracker/current_position -> multi_avoidance

### 优先级 B

- PC 侧单目深度验证
- 输出深度图 / 伪点云

### 优先级 C

- 双目或 ToF 方案选型
- 真实点云与导航融合

### 优先级 D

- A1 板端轻量深度感知裁剪部署

---

## 八、一句话结论

当前最值得做的不是直接上板端 3D 点云，而是先把现有视觉检测结果稳定接入 ROS 避障链；真正的深度与点云，建议优先放在 PC/ROS 侧验证，再决定是否做板端轻量化迁移。
