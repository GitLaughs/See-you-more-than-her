# 07 架构设计

## 本文定位
这份文档只描述当前代码已经体现出来的系统边界、调试路径和构建关系，不把未来集成方向写成默认现状。

## 当前系统边界
- A1 板端：负责图像采集、SCRFD/YOLOv8 推理、OSD、A1_TEST 和到 STM32 的 UART 控制
- SDK 打包层：负责把最新 app 嵌入 EVB 镜像
- Windows 工具：提供相机预览、COM13 调试、直连 STM32 调试和 A1 中继控制，分布在三套独立入口中

## 主要组成

### 板端运行层
默认上板路径是 `ssne_ai_demo`。这层处理图像、推理、叠加显示，以及通过 A1_TEST 和 UART 控制底盘。

### 镜像打包层
`scripts/build_complete_evb.sh` 会重建 Demo、再重新打包 SDK 镜像，所以最终部署单位是 `output/evb/<timestamp>/zImage.smartsens-m1-evb`（同时更新 `output/evb/latest/zImage.smartsens-m1-evb`）。

### Windows 联调层
`tools/aurora/` 提供视频预览、拍照、COM13 终端和 A1_TEST 手动测试；`tools/PC/` 提供电脑直连 STM32 调试；`tools/A1/` 提供 COM13 → A1_TEST → STM32 中继控制。三套工具默认端口分别为 `6201`、`6202`、`6203`。

## 两条重要控制 / 调试路径

### 直连 STM32
PC 串口 → STM32 UART。主要由 `tools/PC/pc_chassis.py` 负责，适合直接看底盘运动和遥测。

### 经由 A1
PC COM13 → A1_TEST → A1 UART0 → STM32 UART3。主要由 `tools/A1/a1_relay.py` 与 `tools/A1/a1_serial.py` 负责，适合联调板端程序和底盘联动。

## 通信边界
- Aurora 只保留视频、拍照、COM13 终端和 A1_TEST 手动测试
- PC 工具负责直连 STM32 路径使用当前桌面端实现的控制协议

## Aurora 相机边界
- Aurora 预览不只是 OpenCV
- 当前链路会用 `qt_camera_bridge.py` 和 QtMultimedia 处理相机源
- 当前默认流程是启动脚本先尝试自动拉起 `Aurora.exe` 做相机初始化，再由 Companion 接管；如有需要可使用 `-SkipAurora`

## 设计上最容易混淆的点
- `ssne_ai_demo` 可以独立构建，但最终烧录关注的是重打包后的 `zImage`
- Windows 工具负责联调，不负责固件烧录本身
